# Обработка публичных учётных данных

> **Источник истины:** `open-sse/utils/publicCreds.ts`
> **Тесты:** `tests/unit/publicCreds.test.ts`
> **Последнее обновление:** 2026-06-28 — v3.8.40
> **Аудитория:** инженеры, интегрирующие провайдеров, которые поставляют публичные OAuth client_id / client_secret / Firebase Web API-ключи в своих публичных CLI.
> **Статус:** **ОБЯЗАТЕЛЬНО** для всего нового кода, встраивающего идентификаторы апстрима.

## Зачем это существует

- [OAuth 2.0 для нативных приложений (PKCE)](https://developers.google.com/identity/protocols/oauth2/native-app) — OAuth client_id / client_secret для установленных приложений публичны; реальную безопасность обеспечивает PKCE.
- [Ключи API Firebase](https://firebase.google.com/docs/projects/api-keys) — идентификаторы веб-клиентов публичны по своей сути.

OmniRoute должен встраивать эти значения, чтобы пользователи, не настраивающие `.env`, всё равно получали рабочий OAuth-поток из коробки. Без встроенного фолбэка провайдеры Gemini / Antigravity / Windsurf перестают работать для любого пользователя, идущего путём «просто клонировать и запустить».

Однако литеральные значения вроде `AIzaSy…`, `GOCSPX-…`, `…apps.googleusercontent.com` совпадают с шаблонами **GitHub Secret Scanning**, **Semgrep** и похожих сканеров. Каждый релиз превращается в шумный поток ложных срабатываний, push protection блокирует легитимные коммиты, и операторы перестают доверять ленте оповещений.

Помощник `open-sse/utils/publicCreds.ts` решает обе проблемы одновременно:

- Встраивает публичный идентификатор как **XOR-маскированную последовательность байтов** (в исходнике нет шаблонов для сканеров).
- Декодирует в рантайме через `decodePublicCred` / `resolvePublicCred`.
- Распознаёт сырые значения, уже следующие известным префиксам (`AIza`, `GOCSPX-`, `<цифры>-<32hex>.apps.googleusercontent.com`, `Iv1.<hex>`), и пропускает их без изменений, так что пользователи с сырыми значениями в существующих `.env` продолжают работать **без миграции**.

Это **обфускация, а не шифрование.** Любой читающий исходник может восстановить значение — и это нормально, потому что значение публично по своей сути. Единственная цель — избежать совпадений с регулярными выражениями сканеров.

## Обязательный паттерн

### 1. Добавление нового публичного учётного значения

Когда нужно встроить новое значение от апстрима, которое:

- происходит из публичного CLI / десктопного приложения / браузерного бандла, **и**
- апстрим-провайдер документирует (или относится к нему) как к публичному идентификатору клиента, **и**
- сканер шаблонов иначе на нём сработает (`AIza…`, `GOCSPX-…`, `<цифры>-…apps.googleusercontent.com` и т.д.),

…следуйте этому чек-листу:

1. Сгенерируйте маскированную последовательность байтов:

   ```bash
   node --import tsx/esm -e \
     'import("./open-sse/utils/publicCreds.ts").then(m =>
        console.log(JSON.stringify(Array.from(
          Buffer.from(m.encodePublicCred("THE_PUBLIC_VALUE"), "base64")
        ))))'
   ```

2. Добавьте новую запись в `EMBEDDED_DEFAULTS` в `open-sse/utils/publicCreds.ts` с **нейтральным именем ключа** (`<provider>_id`, `<provider>_alt`, `<provider>_fb` и т.п.). **Не** используйте в помощнике имена вроде `client_secret` или `api_key` — эти слова вызывают правила Semgrep generic-secret.

3. Добавьте `keyof typeof EMBEDDED_DEFAULTS` в публичное объединение типов (выводится автоматически).

4. В коде-потребителе замените захардкоженный литерал на:

   ```ts
   // одно переопределение через env
   clientSecret: resolvePublicCred("provider_alt", "PROVIDER_OAUTH_CLIENT_SECRET"),

   // несколько псевдонимов env (побеждает первый непустой)
   clientId: resolvePublicCredMulti("provider_id", [
     "PROVIDER_CLI_OAUTH_CLIENT_ID",
     "PROVIDER_OAUTH_CLIENT_ID",
   ]),

   // без переопределения через env (всегда встроенное значение по умолчанию)
   firebaseApiKey: resolvePublicCred("provider_fb"),
   ```

5. Удалите литерал из `.env.example` (замените документацией в комментариях, указывающей читателей сюда):

   ```dotenv
   # ── Provider (Google / Firebase / etc.) ──
   # Публичные OAuth-учётные данные вшиты в код через
   # open-sse/utils/publicCreds.ts. Задавайте эти переменные, только чтобы использовать свои.
   # PROVIDER_OAUTH_CLIENT_ID=
   # PROVIDER_OAUTH_CLIENT_SECRET=
   ```

6. Обновите `tests/unit/publicCreds.test.ts`, добавив проверку формы для нового ключа (проверяйте формат, а не литеральное значение — см. существующие тесты).

7. **Никогда** не добавляйте литералы `AIza…` / `GOCSPX-…` / `…apps.googleusercontent.com` в тестовые файлы. Используйте константы `FAKE_*`, собранные из фрагментов `.join("")` (см. существующие тесты).

### 2. Потребители

- **Читайте только через `resolvePublicCred()` / `resolvePublicCredMulti()`** — никогда не вызывайте `decodePublicCredBytes()` напрямую вне помощника.
- Помощник намеренно дешёв (линейный XOR по байтам) и безопасен для вызова при загрузке модуля; значения по умолчанию вычисляются один раз.
- Переопределение через env всегда побеждает. Если пользователь установил `PROVIDER_OAUTH_CLIENT_SECRET=GOCSPX-myown`, помощник пропускает это сырое значение как есть.

### 3. Запрещённые паттерны

❌ **Никогда** не делайте ничего из перечисленного в продакшн-коде (`src/`, `open-sse/`, `electron/`, `bin/`):

```ts
// ПЛОХО: литеральное значение вызывает Secret Scanning + Semgrep
clientSecret: process.env.PROVIDER_OAUTH_CLIENT_SECRET || "GOCSPX-realvalue",

// ПЛОХО: base64 литерала — GitHub всё равно детектирует с февраля 2025
clientSecret: process.env.PROVIDER_OAUTH_CLIENT_SECRET ||
  Buffer.from("R09DU1BYLXJlYWx2YWx1ZQ==", "base64").toString(),

// ПЛОХО: конкатенация строк, собирающая шаблон в рантайме
clientSecret: "GO" + "CS" + "PX-" + "realvalue",

// ПЛОХО: hex/ROT13-кодирование — другая обфускация, тот же риск детектирования
clientSecret: hexDecode("474f4353..."),
```

Всё это рано или поздно сработает на сканере. Используйте `resolvePublicCred()`.

❌ **Никогда** не добавляйте литеральные учётные данные в `.env.example`. Пользователи, которым нужны реальные значения апстрима, могут извлечь их из публичного CLI сами или использовать собственную регистрацию OAuth.

❌ **Никогда** не отклоняйте новое оповещение secret-scanning, не проверив сначала, не следует ли перенести это значение в данный помощник.

## Связанные механизмы контроля

- `RAW_VALUE_PATTERN` в `publicCreds.ts` перечисляет префиксы, вызывающие прямое пропускание (обратная совместимость). Расширяйте его только для документированных публичных форматов, никогда — для проприетарных секретов.
- `.env.example` проверяется в CI скриптом `check-env-doc-sync` — когда вы удаляете переменную здесь, убедитесь, что документация совпадает.
- Наборы `npm run test:vitest` и `node --import tsx/esm --test tests/unit/publicCreds.test.ts` должны оставаться зелёными.

## Когда НЕ использовать этот помощник

Этот помощник **только** для учётных данных, которые:

1. Публично распространяются апстрим-провайдером (бинарник CLI, браузерный бандл, официальная документация).
2. Документированы или явно подразумеваются как неконфиденциальные (защищённый PKCE, веб-ключ Firebase и тому подобное).

Для всего остального — токенов, выдаваемых операторами, по-тенантных секретов, client_secret вашего собственного OAuth-приложения, ключей шифрования, секретов JWT, паролей БД — используйте **только переменные окружения** (`process.env.FOO`, фолбэк `||` на пустое значение / явную ошибку). Их место — в `.env` и в [зашифрованном хранилище учётных данных](./COMPLIANCE.md), а не в исходниках.
