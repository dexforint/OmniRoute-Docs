# Токен CLI Machine-ID

## Обзор

Команды OmniRoute CLI аутентифицируются против локального API управления с помощью
токена `HMAC-SHA256(machine-id, salt)`, отправляемого в заголовке запроса
`x-omniroute-cli-token`.

Это позволяет подкомандам CLI (`omniroute status`, `omniroute providers` и т.д.)
вызывать эндпоинты управления без необходимости вводить JWT или пароль при каждом
вызове.

## Как это работает

1. `getMachineTokenSync()` читает аппаратный machine ID через `node-machine-id`
   (при ошибке возвращает пустую строку, отключая аутентификацию CLI).
2. Вычисляет `HMAC-SHA256(machine_id, salt)` и возвращает полный 64-символьный
   hex-дайджест — детерминированный, необратимый токен, привязанный к этой машине.
3. CLI отправляет токен как `x-omniroute-cli-token` в каждом запросе к
   `http://localhost:<port>/api/...`.
4. Сервер (`src/server/authz/policies/management.ts`) пересчитывает ожидаемый
   токен с той же солью и сравнивает через `timingSafeEqual`, чтобы предотвратить
   извлечение по времени отклика.

## Свойства безопасности

| Свойство                         | Детали                                                                                                                              |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Только loopback**              | Принимается только когда `Host` — `localhost`, `127.0.0.1` или `::1`.                                                               |
| **Сравнение за постоянное время** | `crypto.timingSafeEqual` предотвращает атаки по времени.                                                                            |
| **Необратимость**                | По выводу HMAC нельзя восстановить machine-id.                                                                                      |
| **Нет обхода путей `always`**    | `isAlwaysProtectedPath()` вычисляется до проверки токена CLI. `/api/shutdown` и `/api/settings/database` всегда требуют JWT.        |
| **Не экспортируется**            | Токен никогда не записывается на диск и не логируется.                                                                              |

## Ротация соли

Установите `OMNIROUTE_CLI_SALT`, чтобы сменить производный токен без изменения кода.
После ротации все процессы CLI на этой машине автоматически начнут использовать
новый токен. Полезно после утечки списка процессов, которая могла раскрыть
прежнее производное значение.

```bash
# Постоянная ротация (добавьте в профиль оболочки)
export OMNIROUTE_CLI_SALT="my-secret-salt-2026"

# Проверьте, что используется новый токен
omniroute status
```

Соль по умолчанию: `omniroute-cli-auth-v1`

## Устаревший формат (SHA-256, 32 символа) — всё ещё принимается

До формата HMAC выше CLI выводил свой токен как
`SHA-256(machineId + salt).hex[0..32]` (32-символьный префикс) в
`bin/cli/utils/cliToken.mjs` (`getLegacyCliTokenSync` в `src/lib/machineToken.ts`).

Для обратной совместимости сервер принимает **оба** формата: верификатор строит
`expectedTokens = [getMachineTokenSync(), getLegacyCliTokenSync()]` и сравнивает
входящий заголовок с каждым через `timingSafeEqual`
(`src/server/authz/policies/management.ts` и `src/lib/middleware/cliTokenAuth.ts`).
Таким образом, токен валиден, если совпадает **либо** с 64-символьным HMAC-дайджестом,
**либо** с 32-символьным устаревшим префиксом SHA-256.

**Отказ от механизма:** установите `OMNIROUTE_DISABLE_CLI_TOKEN=true` (в окружении или `.env`),
чтобы полностью отключить механизм токена CLI; тогда весь доступ потребует явного API-ключа.
На многопользовательских хостах это рекомендуется, так как `machine-id` привязан к
устройству (не к пользователю), и другой пользователь на том же хосте может вычислить
тот же токен.
