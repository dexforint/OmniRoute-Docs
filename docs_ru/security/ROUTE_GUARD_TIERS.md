# Уровни защиты маршрутов

## Обзор

Все маршруты управляющего API OmniRoute классифицируются по одному из трёх уровней
защиты. Классификация статична, определена в `src/server/authz/routeGuard.ts` и
вычисляется до запуска любой другой ветки аутентификации.

## Уровни

### Уровень 1 — LOCAL_ONLY

**Применяется:** `isLocalOnlyPath(path)` → проверка loopback-хоста
**Обход:** по умолчанию нет. Узкое исключение для путей из
`LOCAL_ONLY_MANAGE_SCOPE_BYPASS_PREFIXES`, когда запрос несёт валидный
API-ключ с областью `manage` (см. [Исключение manage-scope](#manage-scope-carve-out)).

Эти маршруты запускают дочерние процессы или исполняют код времени выполнения. Их
выставление на не-loopback трафик позволило бы атакующему, получившему валидный JWT
(например, через туннель Cloudflared/Ngrok), запускать процессы — известный класс
CVE ([GHSA-fhh6-4qxv-rpqj](https://github.com/advisories/GHSA-fhh6-4qxv-rpqj)).

**Что такое GHSA-fhh6-4qxv-rpqj (класс атаки):** управляющий/агентный сервер
выставляет эндпоинт, запускающий подпроцесс (`npm install`, `node`, браузер,
прокси, `git`, `tar`, …). Если этот эндпоинт достижим извне хоста — потому что
оператор поместил OmniRoute за туннель nginx/Cloudflare/Tailscale и утёк JWT,
или аутентификация была настроена неверно — атакующий превращает «вызвать API» в
«выполнить команду на хосте» (удалённое выполнение кода). OmniRoute закрывает это,
применяя **проверку loopback-хоста безусловно, до любой проверки аутентификации**,
на каждом маршруте, способном запускать процессы: утечка токена через туннель всё
равно не достигает запуска процессов.

**Полный набор LOCAL_ONLY.** Авторитетный источник —
`LOCAL_ONLY_API_PREFIXES` / `LOCAL_ONLY_API_PATTERNS` в
`src/server/authz/routeGuard.ts`; таблица ниже зеркалирует текущее состояние.
Контрольная точка `check-route-guard-membership` перечисляет каждый `route.ts` под
префиксами, способными запускать процессы, и проваливает CI, если какой-то не
классифицирован как local-only.

| Префикс / шаблон                    | Почему local-only                                                                          | Обходим ли manage-scope?      |
| ----------------------------------- | ----------------------------------------------------------------------------------------- | ----------------------------- |
| `/api/mcp/`                         | MCP-сервер — запускает stdio-мосты + SSE-обработчики                                       | **Да** (единственный)         |
| `/api/cli-tools/runtime/`           | Рантайм CLI-инструментов — исполняет произвольный код плагинов                             | Нет — способен запускать      |
| `/api/services/`                    | Встроенные сервисы (9router/CLIProxy) — `npm install` + spawn                              | Нет — способен запускать      |
| `/dashboard/providers/services/`    | Обратный прокси к UI встроенных сервисов                                                   | Нет                           |
| `/api/copilot/`                     | Неаутентифицированный LLM-драйвер — по умолчанию только CLI                                | Opt-in оператора: manage/admin |
| `/api/tools/agent-bridge/`          | AgentBridge — запускает MITM-сервер + правки DNS                                           | Нет — способен запускать      |
| `/api/tools/traffic-inspector/`     | Traffic Inspector — слушатель http-proxy + системный прокси                                | Нет — способен запускать      |
| `/api/plugins/`, `/api/plugins`     | Плагины — загрузка/исполнение через `worker_threads` + `child_process`                     | Нет — способен запускать      |
| `/api/system/version`               | Автообновление (только POST; GET/HEAD/OPTIONS исключены) — запускает `git checkout` + `npm install` | Нет                  |
| `/api/db-backups/exportAll`         | Запускает `tar` для архива экспорта                                                        | Нет                           |
| `/api/local/`                       | Однокликовые локальные лаунчеры (сегодня Redis) — запускает podman/docker                  | Нет — способен запускать      |
| `/api/headroom/start`, `/stop`      | Жизненный цикл прокси headroom — запускает python CLI / сигналы PID                        | Нет — способен запускать      |
| `/api/oauth/cursor/auto-import`     | `execFile("which", ["cursor"])` перед импортом учётных данных                              | Нет                           |
| `/api/providers/{id}/login` (regex) | Запускает полноценный Playwright Chromium для входа по веб-cookie                          | Нет                           |

**Ответ при нарушении:** `403 LOCAL_ONLY`

#### Исключение manage-scope

Подмножество путей LOCAL_ONLY МОЖЕТ быть также доступно с не-loopback тогда и только
тогда, когда запрос несёт `Authorization: Bearer <api-key>`, чьи
метаданные включают область `manage` (или `admin`). Исключение закрыто
явно по-путево через `LOCAL_ONLY_MANAGE_SCOPE_BYPASS_PREFIXES`, так что
поведение по умолчанию для любого нового пути LOCAL_ONLY остаётся строгим loopback.
Неаутентифицированные запросы и запросы с ключами без manage по-прежнему
отклоняются с `403 LOCAL_ONLY`.

Сегодня единственный обходимый префикс — `/api/mcp/`. `/api/cli-tools/runtime/` и
`/api/services/` намеренно исключены, потому что они могут запускать произвольные
подпроцессы (`npm install`, `node`), что является ровно тем классом CVE, для
предотвращения которого существует уровень LOCAL_ONLY.

| Запрос                                      | Путь                       | Результат           |
| ------------------------------------------- | -------------------------- | ------------------- |
| Не-loopback, без Bearer                     | `/api/mcp/*`               | 403 LOCAL_ONLY      |
| Не-loopback, Bearer с областью `manage`     | `/api/mcp/*`               | Разрешить           |
| Не-loopback, Bearer без области `manage`    | `/api/mcp/*`               | 403 LOCAL_ONLY      |
| Не-loopback, Bearer с областью `manage`     | `/api/cli-tools/runtime/*` | 403 LOCAL_ONLY      |
| Loopback, любой/без Bearer                  | любой LOCAL_ONLY           | Разрешить (gate пропускает) |

#### Руководство и аудит для оператора

Если вы запускаете OmniRoute за обратным прокси или туннелем (nginx, Caddy, Cloudflare
Tunnel, Tailscale, Ngrok), проверка loopback всё равно защищает перечисленные выше
маршруты с запуском процессов — запрос с не-loopback клиентским адресом отклоняется
с `403 LOCAL_ONLY` **до запуска аутентификации**, так что утечка JWT не достигает
spawn. У оператора остаются две обязанности:

- **Не «чините» 403 подделкой клиентского IP под loopback.** Установка
  `X-Forwarded-For: 127.0.0.1` или прокси, переписывающий адрес источника на
  loopback, заново открывает ровно тот класс RCE, который этот уровень закрывает.
  Выставляйте через прокси панель/API — никогда маршруты с запуском процессов.
- **Держите обход manage-scope минимальным.** Обходим только `/api/mcp/`, и только
  с API-ключом области `manage`. `SPAWN_CAPABLE_PREFIXES` никогда не могут быть
  добавлены в список обхода — схема zod их отклоняет, а
  `isLocalOnlyBypassableByManageScope` отказывает им во время выполнения (эшелонированная
  защита) — именно это панель имеет в виду под «нельзя сделать обходимым».

**Аудит доступа** — чтобы убедиться, что ничто извне хоста не достигает этих маршрутов:

- Откройте **Authorization Inventory** на `/dashboard/settings/security`: там отрисован
  живой список префиксов LOCAL_ONLY, какие префиксы обходимы и набор способных запускать
  процессы («нельзя сделать обходимым»), известный на этапе компиляции.
- Поищите в access-логах вашего обратного прокси перечисленные выше префиксы в паре
  с не-loopback клиентским адресом. Любое такое попадание, вернувшее `200` вместо
  `403 LOCAL_ONLY`, означает, что прокси маскирует реальный клиентский IP — исправьте прокси.
- `403 LOCAL_ONLY` в логах OmniRoute для одного из этих путей — это защита,
  работающая как задумано, а не ошибка, которую надо подавлять.

### Уровень 2 — ALWAYS_PROTECTED

**Применяется:** `isAlwaysProtectedPath(path)` → пропуск обхода `requireLogin=false`
**Обход:** нет при `requireLogin=false`; JWT требуется всегда

Эти маршруты деструктивны или необратимы. Их включение в установке «без пароля»
означало бы, что любой в той же LAN может стереть базу данных или убить
процесс сервера.

| Путь                     | Причина                           |
| ------------------------ | --------------------------------- |
| `/api/shutdown`          | Завершает процесс сервера         |
| `/api/settings/database` | Экспорт, импорт и очистка базы данных |

**Ответ при нарушении:** `401 Authentication required`

### Уровень 3 — MANAGEMENT (по умолчанию)

Все остальные управляющие маршруты. Аутентификация требуется, если не настроено
`requireLogin=false`. Токены CLI могут аутентифицировать эти маршруты (loopback + валидный HMAC).

## Порядок вычисления

```
managementPolicy.evaluate(ctx)
  1. isLocalOnlyPath(path)?
     → loopback                                  → провалиться дальше
     → не-loopback, Bearer manage-scope
        И isLocalOnlyBypassableByManageScope     → разрешить (management_key)
     → иначе                                      → отклонить 403 LOCAL_ONLY
  2. isInternalModelSyncRequest(ctx)?
     → разрешить (system)
  3. hasValidCliToken(headers)?
     → разрешить (cli) [loopback + проверка HMAC через timingSafeEqual]
  4. isAlwaysProtectedPath(path) или requireLogin=true?
     → isDashboardSessionAuthenticated?
        → разрешить (dashboard_session)
     → Bearer manage-scope на необходимом пути?
        → разрешить (management_key)
     → отклонить 401/403
  5. requireLogin=false?
     → разрешить (anonymous)
```

Ветка manage-scope шага 1 — единственный аутентифицированный путь, который может
удовлетворить маршрут LOCAL_ONLY; режим отказа auth-бэкенда возвращает 503 (а не 403),
чтобы истёкшая БД молча не понижала до «запретить».

## Добавление нового маршрута с запуском процессов

1. Добавьте префикс пути в `LOCAL_ONLY_API_PREFIXES` в
   `src/server/authz/routeGuard.ts`
2. Добавьте тест в `tests/unit/authz/routeGuard.test.ts`, проверяющий, что
   `isLocalOnlyPath()` возвращает true для нового префикса
3. **Никогда не пропускайте этот шаг** — см. Жёсткое правило №15 в `CLAUDE.md`
4. Решите: должен ли этот маршрут ТАКЖЕ находиться в `LOCAL_ONLY_MANAGE_SCOPE_BYPASS_PREFIXES`?
   Ответ по умолчанию — **нет**. Включайте только когда маршрут безопасно выставлять
   держателю manage-scope (т.е. он НЕ запускает произвольный пользовательский код).

## Добавление пути, обходимого через manage-scope

1. Убедитесь, что маршрут не исполняет пользовательский код или команды. Если
   исполняет — остановитесь: это исключение — неправильный инструмент.
2. Добавьте префикс в `LOCAL_ONLY_MANAGE_SCOPE_BYPASS_PREFIXES` в
   `src/server/authz/routeGuard.ts`
3. Добавьте покрытие в `tests/unit/authz/management-policy.test.ts` для всех четырёх
   форм запроса: без Bearer (403), Bearer manage (разрешить), Bearer не-manage
   (403), и по-префиксная регрессия, что `/api/cli-tools/runtime/*` остаётся
   строгим loopback даже с Bearer manage.

## Документирование уровней безопасности в OpenAPI

При добавлении нового маршрута в `docs/openapi.yaml` примените соответствующее
вендорное расширение, если маршрут классифицируется `routeGuard.ts`:

| Классификация routeGuard.ts   | Аннотация YAML             | Принуждение                                     |
| ----------------------------- | -------------------------- | ----------------------------------------------- |
| `LOCAL_ONLY_API_PREFIXES`     | `x-loopback-only: true`    | Заблокирован с не-loopback безусловно           |
| `ALWAYS_PROTECTED_API_PATHS`  | `x-always-protected: true` | Аутентификация требуется даже при `requireLogin=false` |
| Внутренний админ/отладочный маршрут | `x-internal: true`  | Скрыт от /dashboard/api-endpoints по умолчанию  |
| Нет (публичный / стандартная auth) | (аннотация не нужна)  | Стандартный доступ, управляемый `requireLogin`  |

### Валидация

Два скрипта обеспечивают согласованность между аннотациями YAML и `routeGuard.ts`:

- `scripts/check/check-openapi-coverage.mjs` — проваливается при покрытии < 99%
- `scripts/check/check-openapi-security-tiers.mjs` — проваливается, если аннотации `x-loopback-only` или
  `x-always-protected` расходятся с константами времени компиляции

Оба скрипта запускаются в pre-commit хуке и в CI.

### Правило ложных срабатываний

Если `x-always-protected` или `x-loopback-only` аннотирован на маршруте, которого НЕТ в
константе `routeGuard.ts`, скрипт покрытия проваливается. Исправление всегда состоит в
выравнивании YAML под то, что `routeGuard.ts` действительно применяет — а не в добавлении
маршрутов в `routeGuard.ts` без реализации логики принуждения.
