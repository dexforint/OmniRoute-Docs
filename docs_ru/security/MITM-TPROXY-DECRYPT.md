# Прозрачная расшифровка MITM TPROXY

Прозрачная расшифровка TPROXY — это **5-й режим захвата** OmniRoute для
MITM-стека [Traffic Inspector](../frameworks/TRAFFIC_INSPECTOR.md) / [AgentBridge](../frameworks/AGENTBRIDGE.md).
Он перехватывает и **расшифровывает** локальный исходящий HTTPS-трафик в Linux,
используя ядерный TPROXY + маршрутизацию по политикам — **без** подделки `/etc/hosts`
и **без** изменения общесистемных настроек прокси. Он дружелюбен к headless
(нет правок DNS, которые нужно убирать), а правила брандмауэра автоматически
сбрасываются при перезагрузке.

В отличие от других режимов захвата, TPROXY не требует настройки для каждого хоста: он
прозрачно перехватывает **произвольные** целевые хосты на заданном порту, завершает TLS
листовым сертификатом, выпускаемым на лету для каждого SNI-имени, захватывает расшифрованный
обмен и повторно шифрует запрос к оригинальному назначению.

> **Только Linux, только root, opt-in.** Этот режим требует Linux, нативный аддон,
> собранный C-инструментарием, и capability **CAP_NET_ADMIN** (обычно root). Он закрыт
> loopback-only API AgentBridge и по умолчанию выключен. Доверенный MITM CA, способный
> подписывать любой хост, — мощная возможность — см. [§6 Безопасность](#6-security).

**Исходный код:** `src/mitm/tproxy/`
**API-маршрут:** `GET / POST / DELETE /api/tools/agent-bridge/tproxy`
**Переключатель в панели:** Traffic Inspector → панель режимов захвата → **"TPROXY Decrypt"** ⚠
**См. также:** [`docs/frameworks/TRAFFIC_INSPECTOR.md`](../frameworks/TRAFFIC_INSPECTOR.md),
[`docs/frameworks/AGENTBRIDGE.md`](../frameworks/AGENTBRIDGE.md)

## §1 Что это и когда использовать

У остальных четырёх режимов захвата есть по ограничению:

| Режим             | Как направляется трафик                    | Ограничение                            |
| ----------------- | ------------------------------------------ | -------------------------------------- |
| AgentBridge       | Подделка `/etc/hosts` DNS для фиксированного набора хостов | только зарегистрированные хосты IDE-агентов |
| Custom Hosts      | Подделка `/etc/hosts` DNS на хост          | одна запись на хост; sudo для правки hosts |
| HTTP_PROXY        | переменные `HTTP_PROXY`/`HTTPS_PROXY`      | только приложения, уважающие эти переменные |
| Общесистемный прокси | настройки прокси ОС                      | изменяет глобальное состояние; требует отката |

Прозрачная расшифровка TPROXY направляет трафик на уровне **ядра**. Она
маркирует новые локальные исходящие TCP-соединения к целевому порту (по умолчанию `443`) в
цепочке `mangle OUTPUT`, `ip rule` перенаправляет маркированные пакеты на локальную доставку,
и при повторном входе цель `TPROXY` в `mangle PREROUTING` передаёт их
слушателю **IP_TRANSPARENT** — который затем завершает TLS и захватывает открытый текст.

Используйте его, когда нужно захватить и расшифровать трафик от процесса, который:

- общается с хостом, не зарегистрированным AgentBridge, и
- не уважает `HTTP_PROXY`, и
- вы не хотите беспокоить общесистемной сменой прокси.

Поскольку перехват происходит в ядре, исходному процессу **не нужна
никакая смена конфигурации** — но процесс должен доверять динамическому CA,
который устанавливает OmniRoute (см. [§4](#4-the-per-sni-dynamic-ca-and-trust-store-installer)).

## §2 Требования

| Требование         | Детали                                                                                                                                            |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ОС**             | Только Linux — **IP_TRANSPARENT** — опция сокета только для Linux. Загрузчик возвращает «unavailable» на любой другой платформе.                  |
| **Привилегии**     | Capability **CAP_NET_ADMIN** для создания прозрачного сокета и применения правил `iptables`/`ip` — на практике, запуск от root.                    |
| **Нативный аддон** | Крошечный N-API аддон (`src/mitm/tproxy/native/transparent.c`) должен быть собран или поставлен как пребилд. См. [§3](#3-the-native-ip_transparent-addon). |
| **Модули ядра**    | `iptables` с поддержкой цели `TPROXY` и совпадений `mangle` и `mark` (проверено на ядре 6.8.0).                                                    |

**Плавная деградация:** если какое-то требование отсутствует (не Linux, нет инструментария,
аддон не собран), загрузчик аддона (`src/mitm/tproxy/transparentSocket.ts::loadTransparentAddon`)
возвращает `null` вместо выброса исключения. Статус режима захвата тогда сообщает
`available: false`, переключатель в панели **отключён** с подсказкой
«TPROXY decrypt requires Linux + root + the native addon», а остальная часть
OmniRoute продолжает работать.

## §3 Нативный аддон IP_TRANSPARENT

Модуль `net` Node не может выполнить `setsockopt(IP_TRANSPARENT)` _до_ `bind()`, что
TPROXY требует (иначе ядро отбрасывает перенаправленные пакеты). Аддон
(`src/mitm/tproxy/native/transparent.c`, собирается через `binding.gyp`) — небольшой N-API
модуль, выставляющий три функции, потребляемые через `transparentSocket.ts`:

| Функция аддона                        | Работа с сокетом                                                                               | Используется для                                                                    |
| ------------------------------------- | ---------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| `createTransparentListener(ip, port)` | `socket()` + **SO_REUSEADDR** + **IP_TRANSPARENT** + `bind()` + `listen()`, возвращает сырой fd | слушатель прозрачного захвата (Node принимает fd через `server.listen({ fd })`)     |
| `setSocketMark(fd, mark)`             | `setsockopt` **SO_MARK** на существующем fd                                                    | анти-петля (маркировка собственных сокетов прокси)                                  |
| `connectMarked(ip, port, mark)`       | `socket()` + **SO_MARK** **до** неблокирующего `connect()`, возвращает fd                       | повторно зашифрованная пересылка апстриму (SYN несёт метку)                         |

Оригинальное назначение читается из `socket.localAddress`/`localPort` — TPROXY
его сохраняет, так что поиска **SO_ORIGINAL_DST**/NAT нет.

### Сборка аддона

```bash
npm run build:native:tproxy      # cd src/mitm/tproxy/native && node-gyp rebuild
                                 # -> native/build/Release/transparent.node
```

- Во время `npm run build` `scripts/build/build-tproxy-native.mjs` запускает `node-gyp
rebuild`. Он **только для Linux и нефатален** — отсутствующий инструментарий просто
  оставляет режим захвата недоступным.
- `assembleStandalone.mjs` копирует `build/Release/transparent.node` в автономный
  бандл; `transparentSocket.ts` резолвит его и относительно модуля, и относительно
  cwd (`<cwd>/src/mitm/tproxy/native/...`).
- `build/` и `prebuilds/` игнорируются git — бинарник **собирается, никогда не коммитится**.

Загрузчик пробует в порядке приоритета:
`native/build/Release/transparent.node`, затем `native/prebuilds/transparent.node`
(оба — относительно модуля и под `<cwd>/src/mitm/tproxy/`).

## §4 Динамический CA по SNI и установщик хранилища доверия

Статический MITM-сертификат AgentBridge работает только потому, что AgentBridge подделывает DNS
для **фиксированного** набора хостов. TPROXY перехватывает **произвольные** хосты, поэтому слушатель
должен предъявить валидный листовой сертификат для любого запрошенного клиентом SNI.

### Динамический CA (`src/mitm/tproxy/dynamicCert.ts`)

`DynamicCertStore` запускает локальный CA (построенный на зависимости `selfsigned`), который:

- Генерирует долгоживущий CA через `generateMitmCa()` (CN `"OmniRoute MITM CA"`,
  действителен 10 лет, `basicConstraints CA=true` + `keyUsage keyCertSign,cRLSign`,
  RSA 2048 бит / SHA-256).
- Выпускает **листовой сертификат по SNI-имени по требованию** через `issueLeafCert()` (действителен
  1 год, `subjectAltName` = SNI-хост) и кэширует один `tls.SecureContext`
  на имя хоста.
- Выставляет `createSNICallback()` для сервера, завершающего TLS (см. [§5](#5-how-decrypt-and-capture-work)).
- Может создаваться с `existingCa`, чтобы CA оставался стабильным между перезапусками
  (тогда хранилище доверия не нужно переустанавливать).

Приватный ключ CA **никогда не покидает машину**.

### Установщик хранилища доверия (`src/mitm/tproxy/caTrust.ts`)

Перехватываемый клиент должен доверять динамическому CA, поэтому запуск режима захвата
устанавливает сертификат CA в хранилище доверия ОС под **выделенным слотом** —
`omniroute-tproxy-ca.crt` (константа `TPROXY_CA_CERT_NAME`) — отдельно от
слота статического MITM-сертификата (`omniroute-mitm.crt`), чтобы они никогда не
перезаписывали друг друга.

`installTproxyCa(caPem, sudoPassword?)` определяет якорный каталог дистрибутива
(по порядку: сначала Debian-стиль) и запускает соответствующую команду обновления:

| Якорный каталог                             | Команда обновления       |
| ------------------------------------------- | ------------------------ |
| `/usr/local/share/ca-certificates`          | `update-ca-certificates` |
| `/etc/ca-certificates/trust-source/anchors` | `update-ca-trust`        |
| `/etc/pki/ca-trust/source/anchors`          | `update-ca-trust`        |
| `/etc/pki/trust/anchors`                    | `update-ca-certificates` |

Установка размещает PEM во временный файл, затем (привилегированно) `mkdir -p` якорный
каталог, `cp` размещённый файл в него и запускает команду обновления. `uninstallTproxyCa()`
удаляет только выделенный слот (не трогая статический MITM-сертификат) и
обновляет — no-op на не-Linux.

Все привилегированные команды выполняются через `execFileWithPassword` (`src/mitm/systemCommands.ts`)
— `spawn` с **массивами аргументов, без shell, без строковой интерполяции** (Жёсткое правило №13).
Когда процесс — root (например, на VPS), цель выполняется напрямую, и пароль не нужен;
на не-root десктопе `sudoPassword` передаётся через `sudo -S` на stdin.

> `sudoPassword` десктопа подаётся в теле POST, чтобы авторизовать установку в
> хранилище доверия; он полностью игнорируется, когда процесс — root.

## §5 Как работают расшифровка и захват

Конвейер (всё под `src/mitm/tproxy/`):

```
локальное приложение ──TCP/443──▶ mangle OUTPUT маркирует соединение (fwmark)
                          ip rule → локальная таблица маршрутов → lo
                          mangle PREROUTING TPROXY → слушатель IP_TRANSPARENT (порт 8443)
                              │  captureMode.ts: читает исходное назначение из socket.localAddress
                              ▼
                          tlsCapture.ts:
                            1. Завершение TLS КЛИЕНТА листовым сертификатом по SNI (dynamicCert)
                            2. внутренний http.Server парсит расшифрованный открытый текст
                            3. захват → globalTrafficBuffer.push() с source: "tproxy"
                               (применены sanitizeHeaders + maskSecret)
                            4. пересылка ПОВТОРНО зашифрованная к оригинальному назначению
                               через сокет с обходной меткой (connectMarked, анти-петля)
                              │
                              ▼
                          оригинальный апстрим (api.example.com)
```

- **Завершение TLS** (`createTlsCaptureServer`): оборачивает сырой перехваченный
  сокет в серверный `tls.TLSSocket`, используя SNI-колбэк динамического CA,
  затем передаёт расшифрованный поток внутреннему `http.Server` (стандартный трюк
  завершения MITM). Время жизни сокетов ограничено `MITM_IDLE_TIMEOUT_MS`, чтобы
  зависший туннель не мог исчерпать файловые дескрипторы.
- **Захват** (`handleDecryptedRequest`): пушит `InterceptedRequest` с
  `source: "tproxy"`, статус начинается как `"in-flight"`, заголовки проходят через
  `sanitizeHeaders()`, а тела — через `maskSecret()` до попадания в
  буфер. Запись затем обновляется ответом, размерами и задержкой.
- **Повторно зашифрованная пересылка** (`createForward` / `realForward`): повторно шифрует к
  оригинальному назначению. `rejectUnauthorized` по умолчанию **`true`** (безопасно по
  умолчанию) — сертификат апстрима проверяется по SNI/Host, который запросил клиент,
  так что прокси отклоняет ровно то, что отклонил бы оригинальный клиент.

### Анти-петля (SO_MARK)

Поскольку правила маркируют новые локальные исходящие соединения, **собственная**
повторно зашифрованная пересылка прокси обычно была бы перехвачена повторно — бесконечная петля.
Путь пересылки защищается обходной меткой сокета (**SO_MARK**):

- `realForward` открывает свой апстрим-сокет через `connectMarked(ip, port, DEFAULT_BYPASS_MARK)`
  — `DEFAULT_BYPASS_MARK = 0x539` — который устанавливает **SO_MARK** **до** `connect()`,
  так что SYN пересылки несёт обходную метку.
- Правило `mangle OUTPUT` исключает соединения, уже несущие обходную метку
  (`-m mark ! --mark <bypassMark>`), так что пересылка прокси **не** маркируется
  повторно и не входит в TPROXY снова.

> Замечание по реализации: сокет с обходной меткой должен быть установлен на
> `createConnection` агента (`https.request({ createConnection })` молча игнорируется
> при наличии агента), иначе пересылка открыла бы немаркированный сокет, и
> петля вернулась бы. Это было подтверждённое e2e исправление анти-петли.

## §6 Безопасность

| Контроль                         | Детали                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **API только loopback**          | `/api/tools/agent-bridge/tproxy` покрывается префиксом `/api/tools/agent-bridge/` в `LOCAL_ONLY_API_PREFIXES` (`src/server/authz/routeGuard.ts`). Принуждение к loopback выполняется **до** аутентификации (Жёсткие правила №15 + №17) — утечка JWT через туннель не может запустить захват TPROXY, который применяет правила `iptables` и устанавливает CA в хранилище доверия через дочерние процессы.                                    |
| **Выделенный слот CA**           | Динамический CA устанавливается в `omniroute-tproxy-ca.crt`, никогда не перезаписывая статический MITM-сертификат.                                                                                                                                                                                                                                                                                                                     |
| **Ключ CA никогда не покидает хост** | `DynamicCertStore` держит ключ CA в памяти; он не экспортируется.                                                                                                                                                                                                                                                                                                                                                                  |
| **Маскирование секретов**        | `maskSecret()` на телах запросов/ответов и `sanitizeHeaders()` на заголовках выполняются **до** `globalTrafficBuffer.push()`.                                                                                                                                                                                                                                                                                                          |
| **Без интерполяции в shell**     | Все команды `iptables`/`ip`/хранилища доверия выполняются через `execFile`/`execFileWithPassword` с массивами аргументов (Жёсткое правило №13).                                                                                                                                                                                                                                                                                        |
| **Проверка сертификата апстрима** | Повторно зашифрованная пересылка проверяет сертификат апстрима по умолчанию (`rejectUnauthorized: true`).                                                                                                                                                                                                                                                                                                                              |
| **Санитизация ошибок**           | Ответы об ошибках маршрута проходят через `sanitizeErrorMessage()` (Жёсткое правило №12).                                                                                                                                                                                                                                                                                                                                              |

**MITM CA — мощная возможность.** CA, которому доверяет ОС и который может подписывать
любой хост, означает, что всё перехваченное OmniRoute может быть расшифровано. Он закрыт
явным, только локальным режимом захвата TPROXY, по умолчанию выключен, и запись в хранилище
доверия удаляется при остановке режима.

## §7 Транзакционное применение / откат правил брандмауэра

Сбой никогда не должен оставлять правило `mangle` или устаревший маршрут. Построитель команд
(`src/mitm/tproxy/commands.ts`) и исполнитель (`src/mitm/tproxy/setup.ts`) гарантируют,
что **откат — в точности обратное применению, в обратном порядке**.

`applyTproxy(cfg)` выполняет команды применения по порядку; при **любой** ошибке запускает
максимально возможный полный `revertTproxy(cfg)` и повторно выбрасывает — так что брандмауэр
либо полностью применён, либо полностью откачен, никогда — наполовину. `revertTproxy(cfg)`
выполняет обратные команды в обратном порядке и проглатывает ошибки (идемпотентен — безопасно
вызывать безусловно, например, из очистки `repairMitm()` AgentBridge).

`validateTproxyConfig(cfg)` выполняется до любой команды: порты должны быть `1–65535`,
`mark`/`routeTable`/`bypassMark` должны быть положительными целыми, и `bypassMark` должен
отличаться от `mark` (анти-петля).

### Команды применения (по порядку)

```bash
ip rule add fwmark <mark> lookup <routeTable>
ip route add local 0.0.0.0/0 dev lo table <routeTable>
iptables -t mangle -A OUTPUT -p tcp --dport <dport> -m mark ! --mark <bypassMark> -j MARK --set-mark <mark>
iptables -t mangle -A PREROUTING -p tcp --dport <dport> -m mark --mark <mark> -j TPROXY --on-port <onPort> --tproxy-mark <mark>
```

Откат удаляет их в обратном порядке: `PREROUTING -D`, `OUTPUT -D`, `ip route del`, `ip rule del`.

> Рецепт основан на **OUTPUT**, потому что сценарий MITM — это _локальный_ исходящий
> трафик (приложения на том же хосте), который TPROXY только в `PREROUTING` не
> видит — `PREROUTING` видит только транзитный трафик. Цепочка `OUTPUT` маркирует новые
> локальные соединения, `ip rule` перенаправляет их на локальную доставку (`lo`), и
> `PREROUTING` затем назначает их прозрачному слушателю.

## §8 Конфигурация

Запрос запуска (`POST /api/tools/agent-bridge/tproxy`) принимает следующие
поля, валидируемые `StartTproxyBodySchema` (`tproxy/route.ts`). Все необязательны
и откатываются к значениям по умолчанию:

| Поле             | Тип                | По умолчанию | Примечания                                                                                                      |
| ---------------- | ------------------ | ------------ | --------------------------------------------------------------------------------------------------------------- |
| **dport**        | int (1–65535)      | `443`        | Целевой TCP-порт для прозрачного перехвата                                                                      |
| **mark**         | int (≥1)           | `0x2333`     | Метка брандмауэра, устанавливаемая в `OUTPUT`, сопоставляемая `ip rule` + `PREROUTING`                          |
| **onPort**       | int (1–65535)      | `8443`       | Порт, который связывает прозрачный (**IP_TRANSPARENT**) слушатель                                               |
| **routeTable**   | int (≥1)           | `233`        | ID таблицы маршрутизации по политикам с маршрутом `local 0.0.0.0/0`                                             |
| **bypassMark**   | int (≥1, ≠ `mark`) | `0x539`      | Обходная метка сокета (**SO_MARK**), которую прокси ставит на свои апстрим-соединения; исключена в `OUTPUT` (анти-петля) |
| **sudoPassword** | string             | —            | Только не-root десктопы: авторизует установку в хранилище доверия; игнорируется под root                        |

**Переменных окружения** для TPROXY **нет** — вся конфигурация через тело POST
или значения по умолчанию выше.

1. Откройте **Traffic Inspector** (`/dashboard/tools/traffic-inspector`).
2. На панели режимов захвата найдите кнопку **"TPROXY Decrypt"** ⚠
   (`src/app/(dashboard)/dashboard/tools/traffic-inspector/components/CaptureModesToolbar.tsx`).
   - Если она **отключена** с подсказкой «TPROXY decrypt requires Linux + root +
     the native addon», нативный аддон недоступен на этом хосте (не Linux,
     нет инструментария или аддон не собран). См. [§2](#2-requirements) и [§3](#3-the-native-ip_transparent-addon).
3. Нажмите кнопку. Она вызывает `POST /api/tools/agent-bridge/tproxy` через
   `startTproxyCaptureMode()` (`src/lib/inspector/tproxyCaptureApi.ts`), который:
   строит динамический CA, открывает прозрачный слушатель, применяет правила
   брандмауэра и устанавливает CA в хранилище доверия ОС.
4. При работе переключатель становится янтарным и показывает живой счётчик перехватов
   (`· <interceptCount>`). Перехваченные запросы появляются в списке запросов с
   `source: "tproxy"`.
5. Нажмите снова для остановки — `DELETE /api/tools/agent-bridge/tproxy` через
   `stopTproxyCaptureMode()` закрывает слушатель, удаляет CA и откатывает
   правила брандмауэра.

Статус режима захвата (running / available / счётчик перехватов / порт слушателя) приходит
из `GET /api/tools/agent-bridge/tproxy` (`getCaptureStatus()` в
`src/mitm/tproxy/captureManager.ts`). Одновременно работает только **одна** сессия TPROXY —
запуск второй отклоняется с «TPROXY capture mode is already running».

## §10 Устранение неполадок

### Переключатель отключён

Нативный аддон не загружается. Подтвердите: вы на Linux, вы собрали аддон
(`npm run build:native:tproxy`), и процесс может загрузить `transparent.node`.
`isTransparentSocketAvailable()` закрывает переключатель; `GET /api/tools/agent-bridge/tproxy`
возвращает `available: false`, когда аддон отсутствует.

### Ничего не захватывается

- Подтвердите, что перехватываемый процесс действительно подключается к настроенному `dport`
  (по умолчанию `443`).
- Подтвердите, что процесс доверяет динамическому CA. CA установлен под
  `omniroute-tproxy-ca.crt`; приложениям с собственным хранилищем доверия (Firefox/Chrome NSS)
  может потребоваться добавить сертификат и туда.
- Запустите самотест **Diagnose** AgentBridge (см.
  [`AGENTBRIDGE.md`](../frameworks/AGENTBRIDGE.md)) для проверок доверия сертификату /
  здоровья сервера.

### Устаревшие правила брандмауэра после сбоя

`revertTproxy()` — точная обратная применению и идемпотентна. Остановка
режима откатывает правила; если OmniRoute был убит посреди сессии, используйте действие
**Repair** AgentBridge (`POST /api/tools/agent-bridge/repair`), чтобы отменить осиротевшее
системное состояние (подделка DNS, корневой CA, системный прокси). Правила `mangle` TPROXY
и маршрут также автоматически сбрасываются при перезагрузке.

### Бесконечная петля / прокси перехватывает свою пересылку

Это случай анти-петли. Подтвердите, что `bypassMark` отличается от `mark` (валидация
это обеспечивает) и что пересылка использует `connectMarked` (в `realForward` — использует).
См. [§5 Анти-петля](#anti-loop-so_mark).
