# Конфигурация и безопасность CORS

OmniRoute управляет тем, какие **браузерные источники (origins)** могут читать кросс-доменные
ответы, из единого централизованного белого списка. Модель **fail-closed по умолчанию**:
ни один источник не разрешён, пока вы его явно не включите. Эта страница документирует, как
резолвится белый список, что на самом деле открывает `CORS_ALLOW_ALL=true` (и, что важно, чего
**не** открывает), как безопасно настроить разработку и продакшн, и предупреждение времени
выполнения, которое панель показывает при активном подстановочном знаке.

**Источник истины:** `src/server/cors/origins.ts` (`resolveAllowedOrigin`,
`applyCorsHeaders`, `getCorsStatus`). Белый список применяется один раз — в
middleware (`src/server/authz/pipeline.ts`); обработчики отдельных маршрутов сами не
устанавливают `Access-Control-Allow-Origin`.

## Как резолвится источник

Для каждого запроса middleware вычисляет значение `Access-Control-Allow-Origin`
в следующем порядке:

1. **`CORS_ALLOW_ALL=true`** (или устаревший `CORS_ORIGIN=*`) → отразить `Origin`
   вызывающего обратно (или `*`, когда заголовка `Origin` нет), с `Vary: Origin`,
   чтобы кэши работали корректно. Та же контрольная точка `applyCorsHeaders()` также добавляет
   `Vary: Accept-Encoding` к каждому ответу 2xx-с-телом на аутентифицированной по токену
   поверхности `/v1*`/`/v1beta*` (`relaxForTokenAuth`, RFC 9110 §12.5.5, issue #6737), чтобы
   нижестоящие/общие кэши могли корректно различать сжатые и несжатые варианты.
2. Иначе `Origin` запроса нормализуется (нижний регистр, убран завершающий слэш)
   и сопоставляется с **объединённым белым списком**:
   - env **`CORS_ALLOWED_ORIGINS`** — список через запятую, и
   - настройка времени выполнения **`corsOrigins`** (Dashboard → Security → _CORS Allowed
     Origins_), внедряемая через `setRuntimeAllowedOrigins()` из
     `src/lib/config/runtimeSettings.ts`.
3. Нет совпадения → **заголовок `Access-Control-Allow-Origin` не выдаётся**. Браузер
   блокирует кросс-доменное чтение. Это и есть задуманное fail-closed поведение по умолчанию.

| Переменная             | Значение                                                                               |
| ---------------------- | -------------------------------------------------------------------------------------- |
| `CORS_ALLOWED_ORIGINS` | CSV точных разрешённых источников (рекомендуется).                                      |
| `CORS_ALLOW_ALL`       | `true`/`1` → отражать любой источник (подстановочный знак). Только для разработки.     |
| `CORS_ORIGIN`          | Устаревшая. `*` ведёт себя как `CORS_ALLOW_ALL`; одиночное значение добавляется в белый список. |

## Модель угроз — что на самом деле открывает `CORS_ALLOW_ALL=true`

Общее предупреждение OWASP («подстановочный CORS = любой сайт может вызвать ваш API»)
стоит воспринимать серьёзно, но поверхность атаки OmniRoute **уже общего случая**
из-за одного конкретного факта реализации:

> **Центральный `applyCorsHeaders()` никогда не выдаёт
> `Access-Control-Allow-Credentials`.** Браузер не откроет _кросс-доменный_
> ответ _с учётными данными_ (cookie), если сервер не отправит
> `Access-Control-Allow-Credentials: true`. Общий путь CORS OmniRoute этого никогда
> не делает.

Что это значит для каждой поверхности даже при `CORS_ALLOW_ALL=true`:

| Поверхность                         | Механизм аутентификации   | Эффект подстановочного CORS                                                                                                                                                                                                     |
| ----------------------------------- | ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Панель / MANAGEMENT `/api/*`        | Cookie-сессия             | Источник отражается, но **без `Allow-Credentials`** браузер **блокирует** чтение с учётными данными. Вредоносный кросс-доменный сайт **не может прочитать** ваши аутентифицированные ответы панели, и cookie сессии не раскрывается. |
| Клиентский API `/v1/*`, `/v1beta/*` | Bearer / заголовок `x-api-key` | Уже разрешительно **по дизайну** (`relaxForTokenAuth`): браузеры никогда сами не прикрепляют `Authorization`/`x-api-key`, поэтому страница атакующего не может предоставить ваш ключ. `CORS_ALLOW_ALL` это не расширяет.      |
| Публичное только-чтение (`/api/health`, …) | Нет                 | Нечувствительно; подстановочный знак безвреден.                                                                                                                                                                                   |

Таким образом, **остаточная** поверхность `CORS_ALLOW_ALL=true` ограничена: (а)
кросс-доменным **чтением** без учётных данных уже неаутентифицированных данных и (б)
прохождением CORS-**preflight** на управляющих маршрутах — которые всё равно требуют
аутентификации, которую кросс-доменная страница предоставить не может. Это **не** вектор
перехвата сессии или кражи учётных данных на общем пути CORS.

### Единственное настоящее исключение — `/api/v1/agents/`

Маршруты Cloud-Agent (`/api/v1/agents/{health,credentials,tasks,tasks/[id]}`) устанавливают
свои **собственные** заголовки CORS
(`src/lib/cloudAgent/api.ts`, `getCloudAgentCorsHeaders`) и **действительно** выдают
`Access-Control-Allow-Origin: <origin>|*` вместе с
`Access-Control-Allow-Credentials: true`. Это единственная поверхность, где
отражение источника и учётные данные сосуществуют, и она **независима от
`CORS_ALLOW_ALL`**. Эти маршруты требуют управляющей аутентификации
(`requireManagementAuth`); операторы, выставляющие панель за пределы хоста, должны
понимать, что это единственное место, где кросс-доменное чтение с учётными данными
разрешено заголовками ответа. Ужесточение этого места до явного белого списка
отслеживается отдельно от данного руководства по CORS.

## Чек-лист для продакшна

- **Никогда не устанавливайте `CORS_ALLOW_ALL=true` в продакшне.** Оставьте её незаданной.
- Установите **явный** список источников — либо переменной окружения, либо полем на вкладке Security:

  ```bash
  CORS_ALLOWED_ORIGINS="https://app.example.com, https://admin.example.com"
  ```

- Если OmniRoute работает за обратным прокси / туннелем (nginx, Caddy, Cloudflare
  Tunnel, Tailscale), CORS — **не** единственный ваш контроль — защита loopback-маршрутов
  всё ещё защищает маршруты, способные запускать процессы (см.
  [ROUTE_GUARD_TIERS](./ROUTE_GUARD_TIERS.md)). Не подделывайте
  `X-Forwarded-For: 127.0.0.1`, чтобы «исправить» 403; это заново открывает класс RCE,
  который закрывает защита маршрутов.
- Подтвердите состояние времени выполнения: панель показывает **постоянный янтарный баннер**
  в Dashboard → Security → Authorization Inventory всякий раз, когда активен
  `CORS_ALLOW_ALL=true`, а `/api/settings/authz-inventory` возвращает конверт
  `cors: { allowAll, allowedOrigins }`, который могут опрашивать инструменты мониторинга.

## Удобство для разработки — разрешить конкретные локальные источники

Подстановочный знак редко нужен даже при разработке. Разрешите только те dev-серверы, которые используете:

```bash
# Dev-серверы Vite (5173) + Next.js (3000), обращающиеся к локальному OmniRoute
CORS_ALLOWED_ORIGINS="http://localhost:5173, http://localhost:3000"
```

Источники сопоставляются без учёта регистра, завершающий слэш игнорируется, поэтому
`http://localhost:3000` и `http://localhost:3000/` эквивалентны. Тот же CSV можно
задать во время выполнения в **Dashboard → Security → CORS Allowed Origins** без
перезапуска.

## API-ключи vs cookie-сессии

- **Bearer / `x-api-key` (поверхность инференса `/v1/*`):** браузеры никогда не прикрепляют
  их автоматически. CORS здесь не является значимым барьером — барьером является API-ключ —
  поэтому эта поверхность намеренно разрешительна, чтобы браузерные и Electron-клиенты могли
  читать ответы, на которые уже имеют право.
- **Cookie-сессия (панель):** защищена fail-closed поведением по умолчанию **и**
  отсутствием `Access-Control-Allow-Credentials` на общем пути. Держите источники
  управления/панели вне любой разрешительной конфигурации; они должны оставаться
  строго fail-closed.

## Пример: обратный прокси перед OmniRoute

CORS применяется самим OmniRoute, поэтому прокси, как правило, **не** должен добавлять
или переписывать заголовки `Access-Control-*` (двойные заголовки ломают браузеры). Завершайте
TLS и пересылайте — пусть OmniRoute отвечает на preflight:

```nginx
# nginx — пересылка в OmniRoute; НЕ вставляйте Access-Control-* здесь
location / {
    proxy_pass http://127.0.0.1:20128;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    # НЕ устанавливайте X-Forwarded-For в 127.0.0.1 — это обезоруживает защиту loopback-маршрутов.
}
```

Задавайте разрешённые браузерные источники в OmniRoute (`CORS_ALLOWED_ORIGINS` или
вкладка Security), а не в прокси.
