# Манифест плагина провайдера

`open-sse/config/providerPluginManifest.ts` определяет JSON-безопасный контракт плагина
провайдера. `open-sse/config/providerPluginManifestRegistry.ts` привязывает этот
контракт к текущему реестру провайдеров для сайдкаров, таких как Bifrost,
CLIProxyAPI или будущего Go/Rust роутера. TypeScript-реестр остаётся
источником истины, но сайдкары могут потреблять манифест без импорта
кода исполнителей, OAuth-дефолтов, заголовков или состояния окружения процесса.

Тот же манифест доступен по HTTP
`GET /api/v1/provider-plugin-manifest` для сайдкаров, работающих вне процесса.

OmniRoute рекламирует этот URL для Bifrost и CLIProxyAPI через
заголовок запроса `X-OmniRoute-Provider-Manifest-Url`. Установите
`OMNIROUTE_PROVIDER_MANIFEST_URL`, когда сайдкару нужен публичный или контейнерный
сетевой URL вместо локального origin запроса.

## Цель

Сдвинуть метаданные провайдеров в сторону контракта плагина, чтобы горячий путь запроса мог
в итоге принадлежать сайдкару с меньшей задержкой, пока OmniRoute сохраняет
TypeScript-маршрут как гейт политик и фолбэк. Манифест аддитивен: он сам по себе не меняет маршрутизацию запросов.

## Контракт

Манифест содержит:

- id провайдера и алиас
- формат апстрима и имя исполнителя
- тип аутентификации, заголовок аутентификации и опциональный префикс аутентификации
- статические метаданные эндпоинта
- правомочность сайдкара и явные причины, когда провайдер должен оставаться на TS
- JSON-безопасные метаданные моделей, такие как длина контекста, флаги vision/reasoning и
  неподдерживаемые параметры
- теги возможностей, включая `apikey`, `oauth`, `custom-executor`,
  `passthrough-models`, `responses` и `sidecar-candidate`

Манифест намеренно исключает:

- OAuth client secrets и значения секретных дефолтов
- резолв окружения времени выполнения
- заголовки запросов и хелперы публичных учётных данных
- динамические билдеры URL
- функции исполнителей
- внутренности пула сессий

## Использование сайдкаром

Сайдкары должны рассматривать `sidecar.eligible` как консервативный сигнал-кандидат, а не
как безусловное решение о маршрутизации. Первая цель импорта должна быть
API-key, провайдеры со статическим эндпоинтом, использующие дефолтный исполнитель. Провайдеры с
кастомными веб-исполнителями, OAuth/сессионными потоками, динамическими билдерами URL или конфигурацией пула
остаются на пути фолбэка TypeScript, пока сайдкар не реализует эквивалентное
поведение и телеметрия не докажет паритет.

Предлагаемые фазы миграции:

1. Сгенерировать и валидировать манифест плагина провайдера из TS-реестра.
2. Научить Bifrost или CLIProxyAPI импортировать манифест для API-key/статических
   провайдеров.
3. Маршрутизировать правомочные провайдеры через сайдкар за `OMNIROUTE_RELAY_BACKEND`,
   сохраняя включённым фолбэк TS.
4. Продвигать провайдеры только когда частота успеха, p99 задержка, стриминговое поведение
   и обработка неподдерживаемых параметров совпадают с TS-путём.
5. Добавлять сайдкар-нативные плагины для кастомных исполнителей по одному семейству провайдеров за
   раз.

## Почему не встраивать провайдеров напрямую в Next

Next-фронтенд не должен владеть выполнением провайдеров. Он должен вызывать границу API.
Бэкенд затем может решить, использовать ли TypeScript-исполнитель,
Bifrost, CLIProxyAPI или будущий нативный сайдкар. Это сохраняет подпись запроса,
проверки allowlist, политику БД и поведение фолбэка централизованными до любой
передачи сайдкару.
