# Очередь слияния и руководство по ручному Merge-Train

Начиная с v3.8.49 (WS3.2/WS3.4 плана качества/скорости) путь слияния по умолчанию для
проверенных PR в `release/vX.Y.Z` — это **очередь слияния Mergify** (`.mergify.yml`);
**ручной merge-train**, описанный ниже, — это ЗАПАСНОЙ ВАРИАНТ — используется во время инцидентов,
заморозки релизов или если план Mergify Open Source когда-либо изменится.

## Путь по умолчанию: очередь Mergify

1. PR проверен/озеленён кампаниями и одобрен пред-мерж ⭐ владельца
   гейтом (отчёт + решение по каждому пункту — см. `/merge-prs` Шаг 0.75).
2. Владелец (или сессия, действующая по решению владельца) применяет метку **`queue`**.
   Метка ЯВЛЯЕТСЯ одобрением слияния; Mergify только выполняет его.
3. Mergify батчит до 10 PR из очереди, валидирует батч по быстрым гейтам,
   и сливает (squash). Красный батч **бисектится автоматически** — проблемный PR
   изолируется за ~log2(N) ревалидаций и удаляется из очереди; остальные продолжают.
4. После слияния непрерывный workflow release-green валидирует новый tip на push
   и открывает issue атрибуции, если комбинация зарегрессировала (никогда авто-откат).

Защитные ограждения (зеркало `CLAUDE.md` Hard Rules #21/#22):

- **Заморозка релиза открыта** → НЕ маркируйте PR, нацеленные на замороженную ветку; сначала перетаргетьте на
  активную `release/vX+1`.
- **PR другой сессии в полёте** → никогда не помечайте его; только владеющая сессия ставит в очередь
  свою собственную работу.
- Диффы только тестов и PR с меткой `hotfix` уже запускают урезанный CI (см.
  `RELEASE_CHECKLIST.md` → Hotfix Fast-Lane); условия очереди принимают любой
  фактически запущенный набор проверок (`#check-failure=0` + `#check-pending=0`).

## Запасной вариант: ручной merge-train

Используется, когда очередь недоступна. Это кодифицирует практику, которая осушила 33 PR за
один день во время цикла v3.8.47:

1. **Соберите батч** (~10–30 проверенных+одобренных PR). Проверьте коллизии `linked:`
   (одинаковые `tap.testFiles`, одинаковые куски CHANGELOG) и сериализуйте их.
2. **Валидируйте ОДИН РАЗ**: в изолированном worktree от tip релиза смержьте все головы батча
   локально, затем запустите эквивалент релиза
   (`npm run check:release-green`, добавьте `--with-build` перед релизом).
   `scripts/release/merge-train.sh <base> <PR#>…` автоматизирует шаги 1–2 (конфликтующие
   PR выбрасываются, поезд продолжается). Полный режим запускает `npm run test:unit` —
   бокс-тюнингованный раннер (`--test-concurrency=20`), **не** два последовательных 4-ядерных CI
   шарда, которые доминировали фазу примерно на 25% от 16-ядерного бокса (исправлено
   2026-07-18). `--fast` (внутридневные мега-train дренажи, одобрено владельцем 2026-07-18)
   сохраняет каждый статический гейт + vitest, но запускает только node:test файлы, изменённые
   заезжающими PR; полный набор должен всё равно прогоняться хотя бы раз в день на
   накопленном tip (один поезд без `--fast`).
3. **Зелёный** → смержьте PR последовательно (перепроверяя `state,headRefOid` перед каждым —
   PR, чья голова передвинулась, возвращается в проверку). Докажите, что чистый дифф каждого слияния — это
   собственное изменение PR (без авто-резолвов-откатов: аудит `git diff --stat` на
   удаления вне скоупа).
4. **Красный** → бисектите батч пополам (валидируйте каждую половину) вместо ре-валидации
   по одному; отбросьте проблемный PR обратно в очередь проверки с доказательствами.
5. **Никогда**: не мержьте во время заморозки в замороженную ветку; `git stash` где-либо;
   массово перезапускать CI в надежде, что красный позеленеет (правило: красный — это информация).

## Тиринг (почему очередь безопасна только с fast-gates)

- **На PR** (quality.yml fast-gates): TIA-затронутые тесты + полный unit 4-shard +
  vitest + lint bag + typecheck + целостность docs/changelog.
- **На батч/tip** (continuous release-green): `--quick` HARD гейты на каждый push в
  релизную ветку; полные `--with-build --full-ci` прогоны 3×/день.
- **На релиз** (ci.yml на релизном PR): полная матрица вкл. E2E ×9,
  package-artifact + tarball boot-smoke, coverage/ratchets.

Ничего не валидируется меньше, чем раньше — тяжёлая поверхность просто запускается на батч/tip
вместо на PR, что и убирает O(N) round-trip'ы.
