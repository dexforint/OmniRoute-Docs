# Операции с Self-Hosted Runner Box (.113 пул)

Self-hosted пул (метки `self-hosted, omni-release`) работает на машине 16 ГБ по адресу
`192.168.0.113`. Два режима отказа повторялись в дни релизов и до v3.8.49 были
ручной дисциплиной; **скрипт-дворник кодифицирует их** (WS3.3 плана качества):

1. **Осиротевшие temp/work каталоги**, заполняющие диск → ошибки SQLite о переполнении диска в середине задачи.
2. **>4 одновременных раннеров** → задачи, убитые OOM (8 одновременно убивали задачи дважды в
   день релиза v3.8.47; 4 одновременно — проверенный потолок).

## Установка дворника (одноразово, на машине)

```bash
sudo mkdir -p /opt/omniroute-ops
sudo cp scripts/ops/runner-janitor.sh /opt/omniroute-ops/
sudo chmod +x /opt/omniroute-ops/runner-janitor.sh
( sudo crontab -l 2>/dev/null; echo '*/30 * * * * /opt/omniroute-ops/runner-janitor.sh >> /var/log/runner-janitor.log 2>&1' ) | sudo crontab -
```

Что делает каждые 30 мин: подметает остатки temp раннера старше 24ч, оповещает при
≥85% использования корневого диска и оповещает, когда больше потолка раннеров (по умолчанию 4, настраивается
через собственное окружение скрипта) процессов `Runner.Listener` запущено. Оповещения попадают в `/var/log/runner-janitor.log`
с ненулевым выходом (ищите `⚠`).

## Правила эксплуатации

- **Потолок: 4 раннера** на машине 16 ГБ. Раннеры 5–8 остаются STOPPED кроме
  явных внепиковых экспериментов — никогда во время окна релиза.
- Остановка раннера посреди задачи отменяет задачу (наблюдалось вживую): `systemctl stop`
  только когда раннер простаивает (`Runner.Listener` без дочернего `Runner.Worker`).
- VPS .15 только для омологации — никогда не запускает CI-раннеры.
