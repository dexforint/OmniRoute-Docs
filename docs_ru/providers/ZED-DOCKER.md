# Интеграция Zed IDE в окружениях Docker

Когда OmniRoute работает внутри Docker, стандартный процесс «Import from Zed Keychain»
завершается ошибкой, потому что контейнер не может обратиться к демону хранилища ключей
хостовой ОС (libsecret в Linux, Keychain в macOS, Credential Manager в Windows), а каталоги
конфигурации Zed на файловой системе хоста по умолчанию не видны внутри контейнера.

## Почему импорт из хранилища ключей не работает в Docker

Внутри контейнера возникают две блокирующие проблемы:

1. **Изоляция файловой системы** — `isZedInstalled()` ищет `~/.config/zed` (Linux),
   `~/Library/Application Support/Zed` (macOS) или эквивалент в Windows. Эти пути
   находятся на хосте и недоступны без явного монтирования тома.
2. **Изоляция IPC** — даже когда каталог конфигурации смонтирован, нативный модуль
   `keytar` взаимодействует со службой хранилища ключей ОС через Unix-сокет или сеанс
   D-Bus. Ни то, ни другое по умолчанию не пробрасывается в контейнер, поэтому чтение
   учётных данных всегда завершается ошибкой.

OmniRoute определяет окружение Docker по двум эвристикам:

- Наличие файла `/.dockerenv` (создаётся демоном Docker при запуске контейнера).
- Строка `docker` в `/proc/1/cgroup` (Linux cgroup v1).

Когда срабатывает любая из эвристик, маршрут импорта возвращает HTTP 422 с
`zedDockerEnvironment: true` и сообщением, направляющим вас на вкладку ручного импорта токена.

## Использование вкладки ручного импорта токена

1. Откройте **Панель управления → Providers → Zed**.
2. Панель **Manual Token Import** находится под карточкой импорта из хранилища ключей. Когда
   OmniRoute обнаруживает Docker, эта панель автоматически разворачивается после первой
   неудачной попытки импорта из хранилища ключей.
3. Выберите провайдера из выпадающего списка (OpenAI, Anthropic, Google, Mistral, xAI,
   OpenRouter или DeepSeek).
4. Вставьте API-ключ в поле пароля.
5. Нажмите **Import**.

Ключ сохраняется как новое подключение провайдера с именем
`Zed Manual Import (<provider>)`.

## Где Zed хранит API-ключи на хосте

Zed хранит ключи провайдеров ИИ в хранилище ключей ОС под именами сервисов вроде
`zed-openai`, `ai.zed.openai`, `zed-anthropic` и т.д. Чтобы извлечь их для ручного
импорта, смотрите:

**Linux**

```
~/.config/zed/settings.json
```

Раздел `language_models` содержит конфигурации провайдеров. Ключи, сохранённые в хранилище
ключей через UI Zed, не лежат в `settings.json` в открытом виде; извлеките их через
просмотрщик хранилища ключей, например GNOME Keyring / Seahorse, или выполнив:

```bash
secret-tool lookup service zed-openai account api-key
```

**macOS**

```
~/Library/Application Support/Zed/settings.json
```

Записи хранилища ключей можно найти в **Keychain Access.app**, выполнив поиск по `zed`.

## Вариант с монтированием тома (продвинутый)

При желании можно смонтировать каталог конфигурации Zed в контейнер в режиме только чтение.
Это не решает проблему хранилища ключей, но может быть полезно для будущих функций,
читающих несекретные значения конфигурации Zed (например, предпочтения моделей).

```yaml
# фрагмент docker-compose.yml
services:
  omniroute:
    image: omniroute:latest
    volumes:
      # Linux-хост
      - "${HOME}/.config/zed:/host-zed-config:ro"
      # macOS-хост (раскомментируйте вместо этого)
      # - "${HOME}/Library/Application Support/Zed:/host-zed-config:ro"
    environment:
      # В будущем: ZED_CONFIG_PATH=/host-zed-config
      PORT: "20128"
```

Примечание: переопределение через переменную окружения `ZED_CONFIG_PATH` пока не
реализовано. Этот фрагмент приведён как справка на случай появления такой функции.

## API ручного импорта

Эндпоинт ручного импорта также можно вызвать напрямую:

```
POST /api/providers/zed/manual-import
Content-Type: application/json
Authorization: Bearer <management-token>

{
  "provider": "openai",
  "token": "sk-...",
  "label": "My Zed OpenAI key"   // необязательно
}
```

При успехе возвращается:

```json
{ "success": true, "connectionId": "...", "provider": "openai" }
```

## Устранение неполадок

| Симптом                              | Причина                       | Решение                          |
| ------------------------------------ | ----------------------------- | -------------------------------- |
| 422 + `zedDockerEnvironment: true`   | Работа внутри Docker          | Используйте вкладку Manual Token Import |
| 404 + `zedInstalled: false`          | Zed не установлен на хосте    | Установите Zed или используйте ручной импорт |
| 403 + доступ к хранилищу ключей запрещён | ОС запретила доступ к хранилищу | Разрешите доступ в диалоге ОС |
| 404 + сервис хранилища ключей недоступен | `libsecret` отсутствует в Linux | Установите `libsecret-1-dev` |
