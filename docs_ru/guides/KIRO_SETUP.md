# Руководство по настройке Kiro

Это руководство описывает добавление аккаунтов Kiro (ИИ-ассистент для кодирования, размещённый в AWS) в OmniRoute,
с акцентом на одновременную работу нескольких аккаунтов без конфликтов сессий.

---

## Предыстория: почему аккаунты Kiro могут конфликтовать

Бэкенд Kiro использует регистрации клиентов AWS SSO OIDC для отслеживания активных сессий.
Ключевое ограничение: **каждая регистрация OIDC-клиента поддерживает только одну активную
сессию одновременно**. Когда второе устройство или пользователь аутентифицируется с тем же
зарегистрированным клиентом, бэкенд инвалидирует refresh-токен первого аккаунта.

Это тот же механизм, который вызывает проблемы при выполнении `kiro-cli login` на
машине, где уже выполнен вход в другой аккаунт Kiro — новый вход отзывает токен
первого аккаунта.

---

## Как OmniRoute это решает (v3.8.0+)

Начиная с v3.8.0, OmniRoute вызывает `registerClient()` (AWS SSO OIDC) при каждом
импорте подключения Kiro. Это даёт каждому подключению OmniRoute собственную выделенную регистрацию OIDC-
клиента. Поскольку каждая регистрация клиента независима, обновление или
повторная аутентификация одного аккаунта не влияет на refresh-токен любого другого аккаунта.

Изоляция применяется к методам импорта по refresh-токену, а авторизация по API-ключу вообще избегает
OIDC-сессий обновления:

| Метод импорта                                  | Статус изоляции                                                                                  |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| AWS Builder ID / поток device-code IDC         | Изолирован с момента появления потока device-code                                                |
| **Import Token** (ручная вставка refresh-токена) | Изолирован с v3.8.0                                                                            |
| **Социальный вход Google / GitHub**            | Изолирован с v3.8.0                                                                              |
| **Auto-Import** (SQLite kiro-cli)              | Изолирован с v3.8.0 (путь SQLite уже был изолирован; откат в кэш SSO теперь тоже изолирован)     |
| **API Key** (долгоживущий ключ CodeWhisperer)  | Нет сессии обновления; ключ валидируется и хранится как bearer-учётные данные                    |

---

## Заметка о миграции для подключений, созданных до v3.8.0

Подключения, импортированные до v3.8.0, не имеют выделенной регистрации OIDC-
клиента, сохранённой в `providerSpecificData`. Эти подключения продолжают работать, но используют общий
эндпоинт обновления социальной авторизации, что означает, что два таких подключения всё ещё могут инвалидировать друг
друга.

**Чтобы получить изоляцию:** удалите старое подключение из **Dashboard → Providers** и
импортируйте его заново через любой из поддерживаемых потоков импорта. Все вновь созданные подключения
получат собственную регистрацию клиента автоматически.

## Добавление двух аккаунтов Kiro рядом

### Предварительные требования

- OmniRoute v3.8.0 или новее.
- Рабочий аккаунт Kiro (email + пароль, вход через Google или GitHub).
- Опционально второй аккаунт Kiro.

### Шаг 1: импортируйте первый аккаунт

1. Откройте **Dashboard → Providers → Add Provider → Kiro**.
2. Выберите один из вариантов:
   - **Import Token** — вставьте refresh-токен, начинающийся с `aorAAAAAG`.
   - **API Key** — вставьте долгоживущий API-ключ Kiro / CodeWhisperer.
   - **Вход через Google / GitHub** — завершите OAuth-поток в браузере.
   - **Auto-Import** — нажмите кнопку; OmniRoute прочитает учётные данные из
     локальной базы kiro-cli или `~/.aws/sso/cache`.
3. Подключение сохранено. Потоки с refresh-токеном автоматически регистрируют выделенный
   OIDC-клиент. Потоки с API-ключом валидируют ключ в AWS и не хранят refresh-токен.

### Шаг 2: импортируйте второй аккаунт

Повторите шаг 1 для второго аккаунта. Поскольку каждый импорт создаёт отдельную регистрацию OIDC-
клиента, два подключения полностью изолированы.

### Шаг 3: убедитесь, что оба подключения активны

1. **Dashboard → Providers** — оба подключения Kiro должны показывать статус **Active**.
2. **Dashboard → Health** — оба подключения должны проходить проверку здоровья токена.

### Шаг 4: используйте комбо для маршрутизации между аккаунтами

Создайте комбо с обоими подключениями в качестве целей для балансировки нагрузки или переключения между ними:

```
kiro/kiro-dev → kiro/kiro-pro
```

Настройку комбо см. в [FEATURES.md](./FEATURES.md) и документации по маршрутизации.

## Корпоративные пользователи / IDC

Для аккаунтов AWS IAM Identity Center (IDC) используйте поток **AWS Builder ID / IDC device-code**
из **Dashboard → Providers → Kiro → Device Code**. Поток device-code
всегда был полностью изолирован. Повторный импорт для этих подключений не требуется.

Корпоративные пользователи, работающие в регионе AWS, отличном от стандартного, могут указать регион при
импорте через API Import Token:

```bash
curl -X POST http://localhost:20128/api/oauth/kiro/import \
  -H "Content-Type: application/json" \
  -d '{"refreshToken": "aorAAAAAG...", "region": "eu-west-1"}'
```

Поле `region` по умолчанию равно `us-east-1`, если не указано.

## Поток импорта по API-ключу

Авторизация по API-ключу предназначена для долгоживущих bearer-учётных данных Kiro / AWS CodeWhisperer. Она
не использует обновление OAuth, поэтому избегает инвалидации общих OIDC-сессий.

### Панель управления

1. Откройте **Dashboard -> Providers -> Kiro**.
2. Выберите **API Key**.
3. Вставьте API-ключ и опциональный регион AWS (по умолчанию `us-east-1`).
4. OmniRoute валидирует ключ и сохраняет подключение.

### API

```bash
curl -X POST http://localhost:20128/api/oauth/kiro/api-key \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "kiro_or_codewhisperer_key", "region": "us-east-1"}'
```

### Внутренний контракт

Маршрут API валидирует ключ вызовом `KiroService.validateApiKey()`, который
использует `ListAvailableProfiles` против соответствующего региону эндпоинта CodeWhisperer/Amazon Q
и разрешает `profileArn`.

Сохранённое подключение использует:

```json
{
  "authType": "apikey",
  "providerSpecificData": {
    "authMethod": "api_key",
    "region": "us-east-1",
    "profileArn": "arn:aws:codewhisperer:..."
  }
}
```

Во время выполнения `KiroExecutor.buildHeaders()` отправляет ключ как
`Authorization: Bearer <key>` и добавляет `tokentype: API_KEY`. Вызовы квоты/профиля
используют тот же маркер, чтобы AWS воспринимал bearer как долгоживущий API-ключ, а не
токен доступа OIDC или социальной авторизации.

## Истечение срока OIDC-клиента

Публичные клиенты AWS SSO OIDC обычно истекают через 90 дней
(`clientSecretExpiresAt`). OmniRoute хранит эту метку времени в `providerSpecificData`
для наблюдаемости. Если подключение перестаёт обновляться примерно через 90 дней, импортируйте его
заново, чтобы получить свежую регистрацию OIDC-клиента. Автоматическая повторная регистрация по
истечении отслеживается как будущее улучшение.

У подключений по API-ключу нет истечения OIDC-клиента, потому что они не обновляются
через AWS SSO OIDC.

## Устранение неполадок

### Второй аккаунт постоянно выходит из системы

- Проверьте оба подключения в **Dashboard → Providers** и убедитесь, что каждое показывает непустой
  `clientId` в своём сыром JSON (видно через иконку информации). Если у какого-либо подключения отсутствует
  `clientId`, оно было импортировано до v3.8.0 — импортируйте его заново.

### Импорт падает с «Token validation failed»

- Убедитесь, что refresh-токен начинается с `aorAAAAAG`.
- Убедитесь, что OmniRoute может достучаться до `https://oidc.us-east-1.amazonaws.com` (или настроенного
  региона). Если вы за корпоративным прокси, задайте прокси на уровне провайдера в
  **Dashboard → Settings → Proxies**.

### Импорт по API-ключу падает

- Убедитесь, что ключ — это API-ключ Kiro / CodeWhisperer, а не refresh-токен.
- Убедитесь, что регион AWS соответствует ключу/аккаунту. По умолчанию `us-east-1`.
- Ключ должен иметь возможность вызывать `ListAvailableProfiles`; иначе OmniRoute не сможет
  разрешить необходимый `profileArn`.

По другим проблемам см. основной [TROUBLESHOOTING.md](./TROUBLESHOOTING.md).
