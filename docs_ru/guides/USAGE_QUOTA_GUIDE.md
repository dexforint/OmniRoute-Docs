# Отслеживание использования, квот и расходов

> **TL;DR**: OmniRoute отслеживает использование токенов каждого запроса, вычисляет стоимость, применяет квоты на каждый API-ключ и отображает аналитику в панели управления. Это руководство объясняет, как всё это работает.

**Источники:**

- `open-sse/services/usage.ts` (~70KB) — основное отслеживание использования
- `src/lib/usageAnalytics.ts` (~10KB) — агрегация для панели управления
- `src/lib/db/quotaSnapshots.ts` — исторические данные квот
- `src/lib/db/usage*.ts` — несколько модулей БД, связанных с использованием

---

## Обзор

Каждый запрос, проходящий через OmniRoute, создаёт **запись об использовании**, которая фиксирует:

- **Идентичность**: какой API-ключ, провайдер, модель, комбо
- **Токены**: токены промпта, токены завершения, кэшированные токены, всего
- **Стоимость**: сумма в USD (вычисляется из данных о ценах)
- **Время**: задержка, метки времени начала/конца
- **Статус**: успех, ошибка, ограничен частотой и т.д.

Эти записи агрегируются в **аналитику**, сохраняются как **снимки квоты** и используются для применения **бюджетных лимитов на ключ**.

```
Запрос ──▶ chatCore ──▶ usage.record() ──▶ SQLite
                                  │
                          ┌───────┼───────┐
                          ▼       ▼       ▼
                    analytics  quota   billing
                    (dashboard) (enforce) (export)
```

## Что записывается

Сервис `usage.ts` захватывает **событие использования** для каждого запроса:

| Поле               | Тип     | Источник                                                   |
| ------------------ | ------- | ---------------------------------------------------------- |
| `id`               | string  | UUID, генерируемый при записи                              |
| `apiKeyId`         | string  | API-ключ, инициировавший запрос                            |
| `provider`         | string  | ID провайдера (openai, anthropic и т.д.)                   |
| `model`            | string  | ID модели (gpt-5, claude-opus-4-6 и т.д.)                  |
| `comboId`          | string? | ID комбо, если маршрутизация через комбо                   |
| `promptTokens`     | number  | Из ответа апстрима                                         |
| `completionTokens` | number  | Из ответа апстрима                                         |
| `cachedTokens`     | number  | Токены попадания в кэш (кэширование промптов Anthropic и т.д.) |
| `totalTokens`      | number  | prompt + completion                                        |
| `costUsd`          | number  | Вычисляется из данных о ценах                              |
| `latencyMs`        | number  | Сквозная длительность запроса                              |
| `status`           | enum    | `success`, `error`, `rate_limited`, `timeout`, `cancelled` |
| `errorClass`       | string? | Класс ошибки, если status != success                       |
| `timestamp`        | string  | ISO 8601 UTC                                               |
| `metadata`         | object  | Пользовательские данные, внедрённые плагинами              |

### Откуда берутся токены

Токены извлекаются из ответа апстрим-провайдера в **обработчике ответа**:

```ts
// From open-sse/handlers/chatCore.ts
const response = await providerExecutor.execute(provider, request);
const usage = response.usage || {
  prompt_tokens: 0,
  completion_tokens: 0,
  cached_tokens: 0,
};
```

Для провайдеров, не возвращающих использование (некоторые провайдеры на веб-cookie), OmniRoute **оценивает** токены эвристикой `~4 символа на токен` (см. `open-sse/services/autoCombo/pipelineRouter.ts`).

### Кэшированные токены

OmniRoute отслеживает `cached_tokens` отдельно от `prompt_tokens`, потому что:

- Кэширование промптов Anthropic тарифицирует кэшированные токены по сниженной ставке (10% от обычной)
- Некоторые провайдеры возвращают `cache_read_input_tokens`, которые должны тарифицироваться иначе
- Аналитика может показывать **долю попаданий в кэш** = `cached_tokens / prompt_tokens`

## Вычисление стоимости

Стоимость вычисляется из **данных о ценах**, синхронизированных из LiteLLM (`src/lib/pricingSync.ts`):

| Модель            | Ввод $/1M | Вывод $/1M | Кэш $/1M |
| ----------------- | --------- | ---------- | -------- |
| gpt-5             | $2.50     | $10.00     | —        |
| claude-opus-4-6   | $15.00    | $75.00     | $1.50    |
| claude-sonnet-4-5 | $3.00     | $15.00     | $0.30    |
| gemini-2.5-pro    | $1.25     | $10.00     | —        |

Формула стоимости (`src/lib/usage/costCalculator.ts`):

```ts
cost =
  (prompt_tokens - cached_tokens) * input_price +
  cached_tokens * cached_price +
  completion_tokens * output_price;
```

> **Зачем вычитать кэшированные из промпта?** Кэшированная часть тарифицируется отдельно; взимание входной цены со всего промпта привело бы к пересчёту.

### Синхронизация цен

Данные о ценах автоматически синхронизируются из LiteLLM через эндпоинт `/api/pricing/sync` (запускается встроенной cron-задачей, а не пользовательской переменной окружения):

```bash
# Ручной запуск
curl -X POST http://localhost:20128/api/pricing/sync
```

Для моделей без данных о ценах OmniRoute откатывается к **оценке стоимости** по внутренним средним ставкам (взятым из данных о ценах LiteLLM).

## Агрегация по диапазонам дат

Модуль `usageAnalytics.ts` вычисляет виджеты панели управления из сырых данных использования. Он поддерживает 7 временных диапазонов:

| Диапазон | Окно                        | Сценарий                    |
| -------- | --------------------------- | --------------------------- |
| `1d`     | Последние 24 часа           | Обнаружение всплесков расходов по часам |
| `7d`     | Последние 7 дней            | Недельный обзор             |
| `30d`    | Последние 30 дней           | Месячный биллинг            |
| `90d`    | Последние 90 дней           | Квартальный анализ          |
| `ytd`    | С 1 января текущего года    | Отслеживание годового бюджета |
| `all`    | Всё время                   | Статистика за всё время     |
| `custom` | Пользовательские начало/конец | Аудиты, произвольные запросы |

### Вычисляемые виджеты панели

Для любого диапазона дат слой аналитики вычисляет:

| Виджет                 | Описание                                               |
| ---------------------- | ------------------------------------------------------ |
| **Сводные карточки**   | Всего запросов, общая стоимость, всего токенов, доля успеха |
| **График дневного тренда** | Стоимость + токены по дням, в стеке по моделям     |
| **Тепловая карта активности** | Сетка час дня × день недели, цвет = число запросов |
| **Разбивка по моделям** | Круговая диаграмма стоимости по моделям               |
| **Разбивка по провайдерам** | Столбчатая диаграмма запросов по провайдерам       |
| **Топ API-ключей**     | Таблица топ-10 ключей по стоимости                     |
| **Анализ ошибок**      | Частота ошибок во времени, топовые классы ошибок       |

### Программный доступ

````ts
import { computeAnalytics } from "@/lib/usageAnalytics";

const analytics = await computeAnalytics(
  history,              // записи истории использования
  "7d",                 // диапазон: "1d" | "7d" | "30d" | "90d" | "ytd" | "all" | "custom"
  connectionMap,        // карта подключений провайдеров (connectionId → имя аккаунта)
  {
    startDate: "2025-01-01",  // опционально: для диапазона "custom"
    endDate: "2025-06-01",   // опционально: для диапазона "custom"
  }
);

console.log(analytics.summary.totalCost);   // 12.34 (центов)
console.log(analytics.byModel[0]);           // { model, cost, requests, promptTokens, completionTokens }

## Применение квот

Квота на каждый API-ключ применяется в двух местах:

1. **Мягкий лимит** (`quotaWarnAt`): предупреждение на панели при превышении порога использования
2. **Жёсткий лимит** (`quotaLimit`): запрос отклоняется с HTTP 429 при превышении

### Конфигурация

```ts
// На каждый API-ключ
await updateApiKey(keyId, {
  quotaWarnAt: 5_00,    // $5.00 — показать предупреждение
  quotaLimit: 10_00,    // $10.00 — жёсткая остановка
  quotaWindow: "month", // "day" | "week" | "month" | "all"
});
````

### Поток применения

```
Запрос ──▶ quotaCheck()
              │
              ├── В пределах лимита?  ──▶ разрешить
              │
              └── Сверх лимита?  ──▶ 429 Too Many Requests
                                      с заголовком Retry-After
```

### Снимки квоты

Таблица `quotaSnapshots` хранит **историческое состояние квоты** для анализа трендов:

| Поле        | Описание                         |
| ----------- | -------------------------------- | ------ | ------- |
| `apiKeyId`  | Отслеживаемый ключ               |
| `window`    | "day"                            | "week" | "month" |
| `used`      | Потрачено в этом окне (центов)   |
| `limit`     | Лимит (центов)                   |
| `resetAt`   | Когда окно сбрасывается          |
| `createdAt` | Когда снят снимок                |

Снимки делаются **на каждый запрос** со стоимостью > 0 и используются, чтобы:

- Отображать индикатор прогресса квоты в панели управления
- Показывать 30-дневные графики тренда квоты
- Запускать оповещения, когда использование приближается к лимиту

### Запрос аналитики использования

Данные об использовании доступны через панель управления или инструменты MCP, а не через прямые эндпоинты экспорта REST. Доступная аналитика:

- **`/api/usage/analytics`** — агрегированные метрики использования (группировка по модели, провайдеру, ключу)
- **`/api/usage/quota`** — текущий статус квоты по каждому API-ключу
- **`/api/usage/history`** — журналы истории запросов

## Инструменты MCP

Два инструмента MCP предоставляют данные об использовании агентам (см. `open-sse/mcp-server/tools/`):

| Инструмент              | Описание                                             |
| ----------------------- | ---------------------------------------------------- |
| `omniroute_cost_report` | Генерирует отчёт о расходах по ключу за заданный период |
| `omniroute_check_quota` | Возвращает текущий статус квоты для API-ключа        |

Пример вызова агентом:

```json
{
  "tool": "omniroute_cost_report",
  "args": { "period": "week" }
}
```

## Хранение и очистка

Данные об использовании растут на ~1-10 КБ на запрос. При масштабе это может быть существенно.

### Настройки хранения

Срок хранения истории использования настраивается через Database Settings в интерфейсе или через `/api/settings/database`.

По умолчанию история использования хранится **90 дней**.

### Очистка

Старые записи очищаются `src/lib/db/cleanup.ts`:

- Запускается фоновым cron-процессом
- Удаляет записи из `usage_history` старше настроенного срока хранения `usageHistory`

### Оценка хранилища

| Частота запросов | Хранилище за 30 дней | Хранилище за 90 дней |
| ---------------- | -------------------- | -------------------- |
| 100 запросов/день | ~3MB                | ~9MB                 |
| 1 000 запросов/день | ~30MB              | ~90MB                |
| 10 000 запросов/день | ~300MB             | ~900MB               |
| 100 000 запросов/день | ~3GB              | ~9GB                 |

При очень высоком трафике рассмотрите:

- Сокращение срока хранения через Database Settings
- Использование `aggregated_metrics` вместо сырых записей (только для аналитики)

## Советы по оптимизации расходов

### 1. Используйте подходящую модель

```bash
# Быстрый ответ — используйте дешёвую + быструю
curl -d '{"model":"auto/fast","messages":[...]}'

# Сложная задача — используйте качественную
curl -d '{"model":"auto/smart","messages":[...]}'
```

### 2. Включите кэширование

Кэширование промптов Anthropic экономит **90% на повторяющемся контексте**:

```ts
// Кэширование автоматическое — просто включите тот же большой системный промпт
const response = await openai.chat({
  model: "claude-sonnet-4-5",
  system: longSystemPrompt, // Будет кэширован автоматически
  messages: [{ role: "user", content: "..." }],
});
```

### 3. Используйте сжатие

Сжатие RTK + Caveman экономит **15-95% в сессиях с большим числом инструментов**:

```ts
const config = {
  compression: {
    engine: "rtk",
    intensity: "aggressive",
  },
};
```

### 4. Задавайте квоты на ключ

Всегда задавайте `quotaLimit`, чтобы предотвратить неконтролируемые расходы:

```ts
await updateApiKey(keyId, { quotaLimit: 10_00 }); // лимит $10/мес.
```

### 5. Аудируйте крупнейших потребителей

Используйте панель управления или **`/api/usage/analytics`** для группировки по API-ключу и сортировки по стоимости:

```bash
GET /api/usage/analytics?groupBy=apiKey
```

## Устранение неполадок

### «Расходы выше ожидаемых»

1. Проверьте **`/api/usage/analytics?groupBy=model`** — найдите дорогую модель
2. Проверьте **`/api/usage/analytics?groupBy=apiKey`** — найдите крупного потребителя
3. Убедитесь, что данные о ценах актуальны: `POST /api/pricing/sync`

### «Записи отсутствуют»

- Проверьте настройки хранения БД в Dashboard → Database → Cleanup — старые записи удаляются периодической задачей очистки (`src/lib/db/cleanup.ts`)
- Проверьте ошибки в `src/lib/db/usage*.ts` — сбои записи в БД логируются, но не выводятся
- Убедитесь, что запрос действительно достиг `chatCore` — проверьте маршрутизацию комбо

### «Квота не применяется»

- Проверьте настройку `quotaLimit` ключа
- Убедитесь, что `quotaWindow` задан корректно
- Ищите записи `quotaSnapshots` — они должны создаваться на каждый запрос
