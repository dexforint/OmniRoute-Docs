## Краткий справочник

**Впервые в OmniRoute?** Начните отсюда — это решает 90% проблем:

| Что вы видите         | Что это значит                       | Что делать                                                                                          |
| --------------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------- |
| «Can't connect»       | OmniRoute не запущен                 | Выполните `omniroute` или `docker restart omniroute`                                                |
| «Invalid API key»     | Ваш ключ неверен или истёк           | Скопируйте ключ заново с сайта провайдера                                                           |
| «Rate limit exceeded» | Вы отправляете слишком много запросов | Подождите 1 минуту или используйте `model: "auto"` для автоматического переключения                |
| «Quota exceeded»      | Вы исчерпали бесплатную/платную квоту | Подключите больше провайдеров или используйте бесплатных (Kiro, Pollinations)                      |
| «Slow responses»      | Провайдер занят или далеко           | Используйте `model: "auto/fast"` или подключите более быстрого провайдера (Groq, Cerebras)          |
| «Wrong provider used» | `auto` выбрал другого провайдера     | Это нормально! `auto` выбирает лучшего. Задайте конкретного провайдера через `model: "openai/gpt-4o"` |
| «502 Bad Gateway»     | Провайдер недоступен                 | Подождите и повторите или используйте `model: "auto"` для смены провайдера                          |
| «401 Unauthorized»    | Ваши учётные данные неверны          | Проверьте API-ключ или пройдите OAuth-аутентификацию заново                                         |
| «429 Too Many Requests» | Сработало ограничение частоты      | Подождите 1 минуту или подключите больше провайдеров                                                |

**Всё ещё не решено?** См. [подробное устранение неполадок](#detailed-troubleshooting) ниже или спросите в [Discord](https://discord.gg/U47eFqAXCn).

## Быстрые решения

| Проблема                                            | Решение                                                                                                                                                        |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Первый вход не работает                             | Задайте `INITIAL_PASSWORD` в `.env` (жёстко заданного значения по умолчанию нет)                                                                               |
| Панель открывается на неверном порту                | Задайте `PORT=20128` и `NEXT_PUBLIC_BASE_URL=http://localhost:20128`                                                                                           |
| Логи не записываются на диск                        | Задайте `APP_LOG_TO_FILE=true` и убедитесь, что захват журналов вызовов включён                                                                                |
| EACCES: permission denied                           | Задайте `DATA_DIR=/path/to/writable/dir`, чтобы переопределить `~/.omniroute`                                                                                  |
| Стратегия маршрутизации не сохраняется              | Обновитесь до последнего релиза v3.x (исправление схемы Zod для сохранения настроек вышло в более ранних версиях)                                              |
| Сбой при входе / пустая страница                    | Проверьте версию Node.js — см. раздел [Совместимость с Node.js](#nodejs-compatibility) ниже                                                                    |
| `dlopen` / `slice is not valid mach-o file` (macOS) | Выполните `cd $(npm root -g)/omniroute/app && npm rebuild better-sqlite3 && omniroute` — см. [Пересборка нативного модуля на macOS](#macos-native-module-rebuild) ниже |
| Прокси: «fetch failed»                              | Убедитесь, что конфигурация прокси задана на правильном уровне — см. раздел [Проблемы с прокси](#proxy-issues) ниже                                            |
| Антивирус помещает `README.md` в карантин           | Ложное срабатывание — см. [Ложные срабатывания антивируса](#antivirus-false-positives) ниже                                                                    |

## Ложные срабатывания антивируса

<a name="antivirus-false-positives"></a>

### Avast/AVG помещают `README.md` в карантин с `MD:HttpRequest-inf[Susp]`

**Это ложное срабатывание. Ничего не заражено, и никаких действий не требуется.**

Avast и AVG используют эвристику, которая помечает текстовые/Markdown-файлы, содержащие множество
ссылок, похожих на HTTP-запросы. Файл `README.md` OmniRoute поставляется внутри npm-пакета (он
указан в `package.json` → `files`), поэтому при глобальной установке он попадает в `node_modules/omniroute/README.md` —
а в нём ~15 примеров `http://localhost:20128/...` (эндпоинты
MCP HTTP/SSE, URL A2A `.well-known` и фрагменты `curl`). Такой плотности ссылок
достаточно для срабатывания эвристики.

Если это началось только недавно: характер файла не изменился. В README расширилась
таблица эндпоинтов (добавлены MCP HTTP + SSE + A2A) и появилось больше примеров `curl`, что вывело
её за порог.

Файл — это инертная документация с нулевым исполняемым содержимым. Вы можете безопасно восстановить его
из карантина.

**Что делать:**

1. **Остановите уведомления** — исключите каталог установки в вашем антивирусе
   (Avast: Настройки → Исключения), добавив ваш глобальный путь `node_modules` и/или
   каталог данных OmniRoute (`~/.omniroute/`).
2. **Сообщите о ложном срабатывании** — <https://www.avast.com/false-positive-file-form.php>,
   приложив помещённый в карантин `README.md`. Это исправление, которое помогает всем, поскольку
   проблема — эвристика вендора, чрезмерно реагирующая на текстовый файл.

**Почему мы не «чиним» это со своей стороны:** все примеры — `http://localhost`, а
localhost не может быть `https` без трений с самоподписанными сертификатами. Искажение документации,
чтобы обойти эвристику одного вендора, навредило бы каждому читателю ради удовлетворения ошибки сканера.

## Совместимость с Node.js

<a name="nodejs-compatibility"></a>

### Страница входа падает или показывает ошибку «Module self-registration»

**Причина:** вы используете версию Node.js ниже одобренного OmniRoute безопасного минимума среды выполнения. Самый частый случай — старый патч-уровень Node 22 или 24, который ниже требуемого OmniRoute пропатченного минимума безопасности.

**Симптомы:**

- Страница входа показывает пустой экран или ошибку сервера
- В консоли `Error: Module did not self-register` или похожие ошибки нативных биндингов
- На странице входа отображается **оранжевый предупреждающий баннер** с вашей версией Node, если среда выполнения вне поддерживаемой безопасной политики

**Решение:**

1. Установите поддерживаемый релиз Node.js LTS (рекомендуется Node.js 24.x):
   ```bash
   nvm install 24
   nvm use 24
   ```
2. Проверьте версию: `node --version` должна показывать `v24.0.0` или новее ветки 24.x LTS
3. Переустановите OmniRoute: `npm install -g omniroute`
4. Перезапустите: `omniroute`

> **Поддерживаемые безопасные версии:** `>=22.22.2 <23` или `>=24.0.0 <27`. Node.js 24.x LTS (Krypton) и Node.js 26 полностью поддерживаются.

### macOS: `dlopen` / «slice is not valid mach-o file»

<a name="macos-native-module-rebuild"></a>

**Причина:** после глобальной установки `npm install -g omniroute` нативный бинарник `better-sqlite3` внутри пакета мог быть собран под другую архитектуру или ABI Node.js, чем используется локально. Это часто случается на macOS (и Apple Silicon, и Intel), когда готовый бинарник не соответствует вашей среде.

**Симптомы:**

- Сервер падает сразу при запуске с ошибкой `dlopen`
- Ошибка содержит `slice is not valid mach-o file`
- Полный пример:

```
dlopen(/Users/<user>/.nvm/versions/node/v24.14.1/lib/node_modules/omniroute/app/node_modules/better-sqlite3/build/Release/better_sqlite3.node, 0x0001): tried: '...' (slice is not valid mach-o file)
```

**Решение — пересоберите под вашу локальную среду (понижать Node.js не требуется):**

```bash
cd $(npm root -g)/omniroute/app
npm rebuild better-sqlite3
omniroute
```

> **Примечание:** это перекомпилирует нативный биндинг под вашу локальную версию Node.js и архитектуру CPU, устраняя несоответствие бинарника. Официально поддерживаемый диапазон среды выполнения — **`>=22.22.2 <23` или `>=24.0.0 <27`** (`SUPPORTED_NODE_RANGE` в `src/shared/utils/nodeRuntimeSupport.ts`, синхронизирован с полем `engines` в `package.json`). Node.js 24.x LTS (Krypton) и Node.js 26 полностью поддерживаются с `better-sqlite3` v12.x.

## Проблемы с прокси

<a name="proxy-issues"></a>

### Валидация провайдера показывает «fetch failed»

**Причина:** эндпоинт валидации API-ключа (`POST /api/providers/validate`) раньше обходил конфигурацию прокси, что вызывало сбои в средах, где требуется маршрутизация через прокси.

**Исправление (v3.5.5+):** теперь исправлено. Валидация провайдера проходит через `runWithProxyContext`, автоматически учитывая настройки прокси на уровне провайдера и глобальные.

### Проверка здоровья токена падает с «fetch failed»

**Причина:** фоновое обновление OAuth-токенов не разрешало конфигурацию прокси для каждого соединения.

**Исправление (v3.5.5+):** планировщик проверки здоровья токенов теперь разрешает конфигурацию прокси для каждого соединения перед попыткой обновления. Обновитесь до v3.5.5+.

### SOCKS5-прокси возвращает «invalid onRequestStart method»

**Причина:** на Node.js 22 диспетчер undici@8 несовместим со встроенной реализацией `fetch()` в Node.

**Исправление (v3.5.5+):** OmniRoute теперь использует собственную функцию `fetch()` из undici, когда активен прокси-диспетчер, обеспечивая единообразное поведение. Обновитесь до v3.5.5+.

### MITM-прокси под WSL: десктопные приложения на хосте Windows не перехватываются

**Причина:** MITM-прокси и его CA-сертификат устанавливаются в среду, где работает OmniRoute. Под WSL этой средой является Linux-гость, в то время как ИИ-десктопные приложения (Kiro, Trae, Copilot, Zed, …) работают на хосте Windows. Приложения хоста не доверяют хранилищу сертификатов гостя и не маршрутизируются через системный прокси гостя, поэтому перехват десктопных приложений там не срабатывает.

**Рекомендация:** запускайте OmniRoute нативно на той же ОС, что и десктопные приложения, которые вы хотите перехватывать (Windows для приложений Windows; macOS/Linux аналогично). Сохранение OmniRoute внутри WSL при работе с приложениями хоста требует ручного добавления сгенерированного CA-сертификата в доверенные на хосте Windows и указания сетевых/прокси-настроек каждого приложения хоста на эндпоинт WSL-прокси — неподдерживаемая, хрупкая конфигурация.

## Проблемы с провайдерами

### «Language model did not provide messages»

**Причина:** квота провайдера исчерпана.

**Решение:**

1. Проверьте трекер квот в панели управления
2. Используйте комбо с резервными уровнями
3. Переключитесь на более дешёвый/бесплатный тариф

### Ограничение частоты запросов

**Причина:** квота подписки исчерпана.

**Решение:**

- Добавьте резерв: `cc/claude-opus-4-6 → glm/glm-4.7 → if/kimi-k2-thinking`
- Используйте GLM/MiniMax как дешёвый резерв

### Истёк OAuth-токен

OmniRoute автоматически обновляет токены. Если проблемы сохраняются:

1. Панель управления → Provider → Reconnect
2. Удалите и заново добавьте подключение провайдера

### Kiro с несколькими аккаунтами: второй аккаунт инвалидирует первый

**Причина:** бэкенд Kiro допускает только одну активную сессию на регистрацию OIDC-клиента.
Когда два аккаунта используют один и тот же зарегистрированный клиент (подключения, импортированные до v3.8.0),
обновление токена одного аккаунта инвалидирует refresh-токен другого.

**Исправление (v3.8.0+):** повторно импортируйте затронутые подключения.
Начиная с v3.8.0, каждое новое подключение Kiro, созданное через **Import Token**,
**социальный вход Google/GitHub** или **Auto-Import**, автоматически регистрирует собственный
выделенный OIDC-клиент. Подключение поэтому полностью изолировано, и обновление одного
аккаунта не влияет ни на один другой аккаунт.

Подключения, импортированные _до_ v3.8.0, не имеют регистрации клиента на уровне подключения.
Эти подключения продолжают использовать общий эндпоинт обновления социальной авторизации.
Чтобы получить изоляцию, удалите старое подключение в Панель управления → Providers и добавьте его заново
через любой из трёх потоков импорта.

Полные детали и пошаговые инструкции по добавлению двух аккаунтов Kiro рядом
см. в [`docs/guides/KIRO_SETUP.md`](./KIRO_SETUP.md).

## Проблемы с облаком

### Ошибки облачной синхронизации

1. Убедитесь, что `BASE_URL` указывает на ваш запущенный экземпляр (например, `http://localhost:20128`)
2. Убедитесь, что `CLOUD_URL` указывает на ваш облачный эндпоинт (например, `https://omniroute.dev`)
3. Держите значения `NEXT_PUBLIC_*` согласованными с серверными значениями

### Облачный `stream=false` возвращает 500

**Симптом:** `Unexpected token 'd'...` на облачном эндпоинте для непотоковых вызовов.

**Причина:** апстрим возвращает SSE-нагрузку, а клиент ожидает JSON.

**Обходной путь:** используйте `stream=true` для прямых облачных вызовов. Локальная среда выполнения включает резервное преобразование SSE→JSON.

### Облако говорит «подключено», но «Invalid API key»

1. Создайте свежий ключ в локальной панели управления (`/api/keys`)
2. Выполните облачную синхронизацию: Enable Cloud → Sync Now
3. Старые/несинхронизированные ключи всё ещё могут возвращать `401` на облаке

## Проблемы с Docker

### CLI-инструмент показывается как не установленный

1. Проверьте поля среды выполнения: `curl http://localhost:20128/api/cli-tools/runtime/codex | jq`
2. Для портативного режима: используйте целевой образ `runner-cli` (встроенные CLI)
3. Для режима монтирования с хоста: задайте `CLI_EXTRA_PATHS` и смонтируйте bin-каталог хоста как read-only
4. Если `installed=true` и `runnable=false`: бинарник найден, но не прошёл проверку здоровья

### Быстрая проверка среды выполнения

```bash
curl -s http://localhost:20128/api/cli-tools/codex-settings | jq '{installed,runnable,commandPath,runtimeMode,reason}'
curl -s http://localhost:20128/api/cli-tools/claude-settings | jq '{installed,runnable,commandPath,runtimeMode,reason}'
curl -s http://localhost:20128/api/cli-tools/openclaw-settings | jq '{installed,runnable,commandPath,runtimeMode,reason}'
```

## Отладка

### Включение файлов логов

Задайте `APP_LOG_TO_FILE=true` в вашем файле `.env`. Логи приложения записываются в `logs/`.
Артефакты запросов сохраняются в `${DATA_DIR}/call_logs/`, когда конвейер журналов вызовов
включён в настройках.
Когда захват конвейера включён, задайте `CALL_LOG_PIPELINE_CAPTURE_STREAM_CHUNKS=false`, чтобы не сохранять
полезные данные потоковых чанков, или настройте `CALL_LOG_PIPELINE_MAX_SIZE_KB`, чтобы изменить предельный размер артефакта в КБ.

### Проверка здоровья провайдеров

```bash
# Панель здоровья
http://localhost:20128/dashboard/health

# Проверка здоровья через API
curl http://localhost:20128/api/monitoring/health
```

### Хранилище среды выполнения

- Основное состояние: `${DATA_DIR}/storage.sqlite` (провайдеры, комбо, алиасы, ключи, настройки)
- Использование: таблицы SQLite в `storage.sqlite` (`usage_history`, `call_logs`, `proxy_logs`) + опционально `${DATA_DIR}/call_logs/`
- Логи приложения: `<repo>/logs/...` (когда `APP_LOG_TO_FILE=true`)
- Артефакты журналов вызовов: `${DATA_DIR}/call_logs/YYYY-MM-DD/...`, когда конвейер журналов вызовов включён

Действие **Clean history** на странице Request Logs очищает `call_logs`, устаревшие
`request_detail_logs` и локальный каталог артефактов `${DATA_DIR}/call_logs/`.

## Проблемы с circuit breaker

### Провайдер застрял в состоянии OPEN

Когда circuit breaker провайдера в состоянии OPEN, запросы блокируются до истечения периода охлаждения.

**Решение:**

1. Перейдите в **Dashboard → Settings → Resilience**
2. Проверьте карточку circuit breaker для затронутого провайдера
3. Нажмите **Reset All**, чтобы сбросить все выключатели, или дождитесь истечения охлаждения
4. Перед сбросом убедитесь, что провайдер действительно доступен

### Провайдер постоянно выбивает circuit breaker

Если провайдер повторно входит в состояние OPEN:

1. Проверьте **Dashboard → Health → Provider Health** на предмет паттерна сбоев
2. Перейдите в **Settings → Resilience → Provider Profiles** и увеличьте порог сбоев
3. Проверьте, не изменил ли провайдер лимиты API или не требуется ли повторная аутентификация
4. Просмотрите телеметрию задержек — высокая задержка может вызывать сбои по таймауту

## Проблемы с транскрипцией аудио

### Ошибка «Unsupported model»

- Убедитесь, что используете правильный префикс: `deepgram/nova-3` или `assemblyai/best`
- Убедитесь, что провайдер подключён в **Dashboard → Providers**

### Транскрипция возвращает пустой результат или падает

- Проверьте поддерживаемые форматы аудио: `mp3`, `wav`, `m4a`, `flac`, `ogg`, `webm`
- Убедитесь, что размер файла в пределах лимитов провайдера (обычно < 25 МБ)
- Проверьте валидность API-ключа провайдера в карточке провайдера

## Отладка транслятора

Используйте **Dashboard → Translator** для отладки проблем трансляции форматов:

| Режим            | Когда использовать                                                                                   |
| ---------------- | ---------------------------------------------------------------------------------------------------- |
| **Playground**   | Сравнить входной и выходной форматы рядом — вставьте падающий запрос, чтобы увидеть, как он транслируется |
| **Chat Tester**  | Отправлять живые сообщения и инспектировать полную нагрузку запроса/ответа, включая заголовки          |
| **Test Bench**   | Запускать пакетные тесты по комбинациям форматов, чтобы найти, какие трансляции сломаны                |
| **Live Monitor** | Наблюдать поток запросов в реальном времени, чтобы поймать периодические проблемы трансляции           |

### Частые проблемы форматов

- **Теги thinking не появляются** — проверьте, поддерживает ли целевой провайдер thinking, и настройку бюджета thinking
- **Вызовы инструментов пропадают** — некоторые трансляции форматов могут отбрасывать неподдерживаемые поля; проверьте в режиме Playground
- **Системный промпт отсутствует** — Claude и Gemini по-разному обрабатывают системные промпты; проверьте результат трансляции
- **SDK возвращает сырую строку вместо объекта** — решено в v1.x; очиститель ответов удаляет нестандартные поля (`x_groq`, `usage_breakdown` и т.д.), вызывающие сбои валидации Pydantic в OpenAI SDK. Если вы всё ещё видите это на v3.x+, пожалуйста, заведите issue.
- **GLM/ERNIE отклоняет роль `system`** — решено в v1.x; нормализатор ролей автоматически объединяет системные сообщения в пользовательские для несовместимых моделей. Если вы всё ещё видите это на v3.x+, пожалуйста, заведите issue.
- **Роль `developer` не распознаётся** — решено в v1.x; автоматически преобразуется в `system` для провайдеров не-OpenAI. Если вы всё ещё видите это на v3.x+, пожалуйста, заведите issue.
- **`json_schema` не работает с Gemini** — решено в v1.x; `response_format` теперь преобразуется в `responseMimeType` + `responseSchema` Gemini. Если вы всё ещё видите это на v3.x+, пожалуйста, заведите issue.

## Настройки отказоустойчивости

### Автоматическое ограничение частоты не срабатывает

- Автоматическое ограничение применяется только к провайдерам с API-ключом (не к OAuth/подписке)
- Убедитесь, что в **Settings → Resilience → Provider Profiles** включено автоматическое ограничение частоты
- Проверьте, возвращает ли провайдер коды статуса `429` или заголовки `Retry-After`

### Настройка экспоненциальной задержки

Профили провайдеров поддерживают эти настройки:

- **Base delay** — начальное время ожидания после первого сбоя (по умолчанию: 1 с)
- **Max delay** — максимальный предел времени ожидания (по умолчанию: 30 с)
- **Multiplier** — насколько увеличивать задержку на каждый последующий сбой (по умолчанию: 2x)

### Защита от «thundering herd»

Когда много параллельных запросов попадает на ограниченного по частоте провайдера, OmniRoute использует мьютекс + автоматическое ограничение частоты, чтобы сериализовать запросы и предотвратить каскадные сбои. Для провайдеров с API-ключом это происходит автоматически.

## Опциональная таксономия сбоев RAG / LLM (16 проблем)

Некоторые пользователи OmniRoute размещают шлюз перед RAG- или агентными стеками. В таких конфигурациях часто встречается странный паттерн: OmniRoute выглядит здоровым (провайдеры работают, профили маршрутизации в порядке, нет алертов об ограничении частоты), но итоговый ответ всё равно неверный.

На практике такие инциденты обычно происходят из нижестоящего конвейера RAG, а не из самого шлюза.

Если вам нужен общий словарь для описания таких сбоев, можно использовать WFGY ProblemMap — внешний текстовый ресурс под лицензией MIT, определяющий шестнадцать повторяющихся паттернов сбоев RAG / LLM. На верхнем уровне он охватывает:

- дрейф выборки и нарушенные границы контекста
- пустые или устаревшие индексы и векторные хранилища
- несоответствие эмбеддингов и семантики
- проблемы сборки промпта и контекстного окна
- коллапс логики и излишне уверенные ответы
- сбои длинных цепочек и координации агентов
- память нескольких агентов и дрейф ролей
- проблемы развёртывания и порядка начальной загрузки

Идея проста:

1. Когда вы исследуете плохой ответ, зафиксируйте:
   - задачу пользователя и запрос
   - маршрут или комбо провайдеров в OmniRoute
   - любой RAG-контекст, использованный ниже по потоку (извлечённые документы, вызовы инструментов и т.д.)
2. Сопоставьте инцидент с одним-двумя номерами WFGY ProblemMap (`No.1` … `No.16`).
3. Сохраните номер в вашей панели, runbook или трекере инцидентов рядом с логами OmniRoute.
4. Используйте соответствующую страницу WFGY, чтобы решить, нужно ли менять ваш RAG-стек, ретривер или стратегию маршрутизации.

Полный текст и конкретные рецепты находятся здесь (лицензия MIT, только текст):

[WFGY ProblemMap README](https://github.com/onestardao/WFGY/blob/main/ProblemMap/README.md)

Вы можете пропустить этот раздел, если не запускаете RAG- или агентные конвейеры за OmniRoute.

## Известные проблемы v3.8.0

Проблемы, специфичные для релиза v3.8.0, и их текущие обходные пути. Если исправление выйдет в более позднем патче, запись будет обновлена или удалена.

### OAuth-поток Windsurf падает с 401

**Симптомы:**

- «401 unauthorized» при завершении OAuth-потока Windsurf из панели управления
- Карточка провайдера Windsurf остаётся в состоянии «needs reconnection» после callback

**Причины:**

- Переменная окружения `WINDSURF_FIREBASE_API_KEY` отсутствует или пуста
- `WINDSURF_API_KEY` настроен неверно или указывает на устаревший токен
- Локальный файрвол/прокси блокирует OAuth callback

**Решение:**

1. Убедитесь, что обе переменные `WINDSURF_FIREBASE_API_KEY` и `WINDSURF_API_KEY` заданы в `.env`
2. Перезапустите OmniRoute, чтобы подхватить новые значения окружения
3. Запустите OAuth-поток заново из **Dashboard → Providers → Windsurf → Reconnect**

### Сбои аутентификации Devin CLI

**Симптомы:**

- «Devin CLI not found» или «auth failed» при вызове инструментов на базе Devin
- Проверка среды выполнения CLI сообщает `installed=false`

**Причины:**

- `CLI_DEVIN_BIN` указывает на несуществующий путь
- Devin CLI не установлен на хосте

**Решение:**

1. Установите Devin CLI для вашей платформы
2. Задайте `CLI_DEVIN_BIN=/usr/local/bin/devin` (или реальный путь) в `.env`
3. Перезапустите OmniRoute и протестируйте заново из **Dashboard → CLI Tools**

### Охлаждение модели зависло (ручной сброс)

**Симптомы:**

- Модель остаётся в списке охлаждения даже после истечения времени
- Запросы по-прежнему пропускают модель в маршрутизации комбо, несмотря на то что метка времени в прошлом

**Ручной сброс:**

- **Панель управления:** **Settings → Model Cooldowns** → нажмите **Re-enable** на затронутой карточке
- **API:** `DELETE /api/resilience/model-cooldowns` с заголовками управленческой авторизации

### Подключение провайдера Command Code падает с 403

**Симптомы:**

- 403 при тестировании подключения провайдера Command Code
- Карточка провайдера показывает «unauthorized» после свежего добавления

**Причина:** OAuth-поток не завершился (callback не получен или токен не сохранён).

**Решение:**

- Выполните `omniroute providers` из CLI, чтобы перезапустить OAuth-поток, или
- Запустите OAuth заново из **Dashboard → Providers → Command Code → Reconnect**

### ModelScope возвращает агрессивные охлаждения 429

**Симптомы:**

- Очень короткие или мгновенные охлаждения на ModelScope после небольшого всплеска запросов
- Маршрутизация комбо пропускает ModelScope раньше, чем ожидалось

**Причина:** ModelScope выдаёт специфичные для провайдера заголовки `Retry-After`. В v3.8.0 добавлена специальная обработка этих заголовков, поэтому более старые версии ошибочно читают их как общие признаки ограничения частоты.

**Решение:**

- Убедитесь, что у вас v3.8.0 или новее
- Убедитесь, что переключатель `useUpstream429BreakerHints` включён в **Settings → Resilience**

### OMNIROUTE_WS_BRIDGE_SECRET отсутствует в продакшене

**Симптомы:**

- 401 на каждый запрос WebSocket-моста Codex/Responses при запуске на удалённом продакшен-хосте
- Рукопожатие WebSocket-моста закрывается сразу после подключения

**Причина:** переменная окружения `OMNIROUTE_WS_BRIDGE_SECRET` отсутствует в продакшен-окружении.

**Решение:**

1. Сгенерируйте случайный секрет: `openssl rand -hex 32`
2. Задайте `OMNIROUTE_WS_BRIDGE_SECRET=<random-secret>` в окружении продакшен-сервера (и в любом клиенте, который обращается к мосту)
3. Перезапустите OmniRoute

### Responses API: фоновый режим деградирован до синхронного

**Симптомы:**

- В логах предупреждение: `background mode degraded to synchronous`
- Запрос с `background: true` возвращает обычный синхронный ответ вместо дескриптора фоновой задачи

**Причина:** v3.8.0 намеренно деградирует `background: true` в Responses API до синхронного выполнения, выдавая предупреждение. Полноценное асинхронное фоновое выполнение — задача будущих релизов.

**Решение:**

- Настройте клиент на вызов без `background`, или
- Дождитесь более позднего релиза с полным асинхронным фоновым режимом (следите за changelog)
