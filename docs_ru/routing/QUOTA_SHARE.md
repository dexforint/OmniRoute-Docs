# Движок распределения квот (Quota Sharing Engine)

> **Ссылка на документ**: `docs/routing/QUOTA_SHARE.md`
> Часть Группы B (планы 16 + 22).

---

## Обзор

Движок распределения квот честно распределяет временную квоту провайдера
(например, 5-часовое окно Codex, Kimi 1500 запросов/час) между несколькими
API-ключами, использующими одно и то же подключение.

**Проблема, которую он решает:** OmniRoute проксирует множество API-ключей к
одному аккаунту апстрим-провайдера. Без логики распределения всплеск от ключа A
может исчерпать часовую квоту провайдера, оставив ключи B и C заблокированными
до сброса окна. Движок предотвращает это, выполняя следующее:

1. Отслеживание скользящего потребления каждого ключа по каждому измерению (%, запросы, токены, $).
2. Применение алгоритма честного распределения с сохранением работы
   (work-conserving fair-share): ключ может занимать из простаивающих долей,
   пока глобальный пул не насыщен.
3. Применение результата в горячем пути (`chatCore.ts`) до того, как запрос
   достигнет апстрим-исполнителя.

## Алгоритм: Fair-Share Work-Conserving

Реализован в `src/lib/quota/fairShare.ts`.

### Режимы

| Условие                                  | Режим          | Поведение                                                  |
| ------------------------------------------ | ------------ | ---------------------------------------------------------- |
| `globalUsedPercent < saturationThreshold`  | **Щедрый (generous)** | Ключ может занимать до глобального лимита минус суммарное потребление |
| `globalUsedPercent >= saturationThreshold` | **Строгий (strict)**  | Жёсткое применение индивидуальной честной доли              |

По умолчанию `saturationThreshold = 0.5` (переменная окружения `QUOTA_SATURATION_THRESHOLD`).

### Решение по каждому измерению

Для каждого активного измерения в пуле движок вычисляет:

```
fairShareAllowed = poolLimit × (allocationWeight / 100)
consumed        = текущее скользящее значение для этого ключа (из QuotaStore.peek)
remaining       = fairShareAllowed - consumed
```

Далее:

- **`policy = hard`**: если `consumed > fairShareAllowed` и режим строгий → **блокировка**.
- **`policy = soft`**: если `consumed > fairShareAllowed` и режим строгий → **штраф** (понижение приоритета в комбо; жёсткой блокировки нет).
- **`policy = burst`**: разрешать, пока есть глобальный запас, независимо от честной доли.

### Абсолютный потолок (cap absoluto)

`capValue` + `capUnit` у аллокации — жёсткий потолок, не зависящий от режима и
политики. Любое измерение, где `consumed >= capValue`, всегда **блокирует** запрос.

### Проверка по нескольким измерениям

Запрос блокируется, если **хотя бы одно** измерение в пуле его блокирует.
Измерения независимы — исчерпание 5h% не влияет на измерение weekly%.

### Заимствование

В щедром режиме ключ, чья аллокация недопотреблена, может использовать
избыток из незанятых долей других ключей. Формула:

```
maxAllowed = globalLimit - consumedByOtherKeys
```

где `consumedByOtherKeys = consumedTotal - consumedByThisKey`. Глобальный
потолок (teto global — `limit` пула для этого измерения) — всегда жёсткий максимум.

## Счётчик со скользящим окном

Реализован в `src/lib/quota/sqliteQuotaStore.ts` и `redisQuotaStore.ts`.

Два сегмента (bucket) на пару `(apiKeyId, dimensionKey)`:

- `curr`: текущий сегмент (`floor(nowMs / windowMs)`)
- `prev`: предыдущий сегмент (`curr - 1`)

Эффективное скользящее значение:

```
effectiveBucketIndex = floor(nowMs / windowMs)
bucketStartMs        = effectiveBucketIndex × windowMs
elapsed              = nowMs - bucketStartMs
weight               = 1 - elapsed / windowMs

effective = prev × weight + curr
```

**Точность**: ~99%. Погрешность не превышает 1% от размера окна на границе
между сегментами (свойственно двухсегментной аппроксимации).

### Конкурентность

Драйвер SQLite: мьютекс в памяти на ключ `(apiKeyId | dimensionKey)`
предотвращает гонку «чтение—изменение—запись». Паттерн повторяет защиту от
thundering-herd в `src/sse/services/auth.ts`.

Драйвер Redis: Lua-скрипт EVAL для атомарного инкремента — выполняется как
одна команда Redis.

## Драйверы

### SQLite (по умолчанию, без установки)

- Таблица: `quota_consumption` (см. миграцию `073_quota_pools.sql` / `074_quota_consumption.sql`).
- Лучший вариант для развёртываний с одним инстансом.
- Всё хранение — в существующей базе SQLite OmniRoute (`DATA_DIR/storage.sqlite`).

### Redis (опционально, мульти-инстанс)

- Требуется npm-пакет `ioredis`.
- Счётчики хранятся в Redis; метаданные (пулы/аллокации) по-прежнему в SQLite.
- Лучший вариант для развёртываний с несколькими репликами, где счётчики должны быть общими.

### Переключение драйверов

Через UI настроек (`/dashboard/settings` → Quota Store) или через переменные окружения:

```bash
QUOTA_STORE_DRIVER=redis
QUOTA_STORE_REDIS_URL=redis://localhost:6379
```

Настройка в БД имеет приоритет над переменными окружения. Если `driver=redis`,
но URL отсутствует или `ioredis` не установлен, фабрика откатывается на SQLite
и пишет предупреждение в лог.

Порядок выбора драйвера:

1. Настройка БД `quotaStore.driver`
2. Переменная окружения `QUOTA_STORE_DRIVER`
3. По умолчанию: `sqlite`

## Несколько измерений

У пула может быть несколько измерений. Каждое измерение независимо:

```ts
QuotaDimension {
  unit: "percent" | "requests" | "tokens" | "usd",
  window: "5h" | "hourly" | "daily" | "weekly" | "monthly",
  limit: number,  // глобальный потолок пула для этого измерения
}
```

**Пример: план Codex** (5h% + weekly%):

```json
[
  { "unit": "percent", "window": "5h", "limit": 100 },
  { "unit": "percent", "window": "weekly", "limit": 100 }
]
```

Запрос должен удовлетворять всем измерениям, чтобы быть разрешённым.

## Интеграция в конвейер

### Хук PRE (`open-sse/handlers/chatCore.ts`)

Выполняется до апстрим-исполнителя, после проверок аутентификации и политик:

```
resolveComboTargets / handleSingleModel
  → enforceQuotaShare(apiKeyId, connectionId, provider, estimatedCost)
      → getQuotaStore().peek() по каждому измерению
      → fairShare.decideFairShare()
      → если блокировка → вернуть 429 (buildErrorBody, Жёсткое правило №12)
      → если разрешено + понижение → установить quotaSoftPenalty=true на кандидате
  → executor.execute()
```

**Fail-open**: если `enforceQuotaShare` выбрасывает исключение, запрос
пропускается с логом `pino.warn`. Это предотвращает блокировку всего трафика
из-за ошибки в движке квот.

### Хук POST (запись потребления)

После успешного ответа:

```
executor возвращает успех
  → spendRecorder.recordConsumption(apiKeyId, connectionId, provider, actualCost)
      → getQuotaStore().consume() по каждому измерению
      → fail-open: ошибки пишутся в pino.warn, клиенту никогда не передаются
```

**Замечание о расхождении**: если `consume` завершается ошибкой после ответа,
скользящий счётчик недосчитывает. Сигнал насыщения от провайдера
(например, `anthropic-ratelimit-unified-5h-utilization`) корректирует
глобальную оценку при следующем запросе.

### Мягкий штраф в комбо (`open-sse/services/combo.ts`)

Когда `decision.deprioritize === true`:

```ts
if (candidate.quotaSoftPenalty) {
  score *= QUOTA_SOFT_DEPRIORITIZE_FACTOR; // по умолчанию 0.7
}
```

Штраф применяется после всех прочих факторов оценки. Он снижает вероятность
выбора насыщенного ключа в авто-комбо без жёсткой блокировки.

## Устранение неполадок

### Redis настроен, но не подключается

Проверьте, что `ioredis` установлен (`npm ls ioredis`) и `QUOTA_STORE_REDIS_URL`
доступен. При ошибке подключения фабрика откатывается на SQLite (записывается в
лог на уровне `warn`).

### `peek` возвращает устаревшие данные / fail-open

Если `peek` выбрасывает исключение, `enforceQuotaShare` трактует результат как
«разрешить» (fail-open). Проверьте логи `pino` на наличие записей `quota:enforce`
и `quota:factory`, чтобы выявить первопричину.

### Расхождение счётчиков потребления

Если фактическое использование провайдера отличается от счётчиков — это
ожидаемо: двухсегментное скользящее окно имеет погрешность ~1% на границах
окон, а `consume` выполняется в режиме fire-and-forget после ответа. Сигнал
насыщения (`saturationSignals.ts`) читает реальную загрузку провайдера с TTL
30 с и соответственно корректирует `globalUsedPercent`.

### Пул показывает «нет данных» для скорости расхода

`computeBurnRate` требует минимум 2 исторических замера. Новые пулы без
предыдущих вызовов `consume` покажут `tokensPerSecond: 0` и
`timeToExhaustionMs: null`.

## Миграция из localStorage

При первой загрузке `/dashboard/costs/quota-share` хук `useLocalStoragePoolMigration`
проверяет:

1. `localStorage.getItem("omniroute:quota-share:pools")` не пусто.
2. `GET /api/quota/pools` возвращает `[]` (БД пуста).

Если оба условия истинны, он пакетно отправляет каждый устаревший пул в
`POST /api/quota/pools`, затем удаляет ключ из localStorage. Миграция
идемпотентна: условие 2 предотвращает повторную миграцию.

## Внутренняя классификация стратегии

`quota-share` — **только внутренняя** стратегия маршрутизации
(`INTERNAL_ROUTING_STRATEGY_VALUES` в `src/shared/constants/routingStrategies.ts`).
Она используется исключительно системно-созданными комбами `qtSd/` и
намеренно исключена из `ROUTING_STRATEGY_VALUES`, чтобы никогда не появляться
как доступная пользователю опция в UI или API.
